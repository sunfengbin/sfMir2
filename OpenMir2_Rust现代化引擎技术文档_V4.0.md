# OpenMir2 Rust现代化引擎 技术文档 V4.0

**文档编号**：OM2-TD-2026-V4.0
**版本状态**：归档版
**编制目的**：基于传统8服务架构 + Bevy客户端 + ECS服务端引擎的技术方案
**核心定位**：服务端与客户端均采用Rust生态，共享技术栈，全 ECS 架构

---

## 文档概述

本文档定义的引擎架构延续传统Mir2传奇引擎的**8服务经典架构**，同时在技术实现层面做出两项根本性升级：

第一，客户端引擎选用**Bevy**（Rust原生游戏引擎），而非传统C++/C#引擎，实现服务端与客户端的**技术栈统一**，代码复用率最大化。

第二，M2Server主引擎采用**ECS（Entity Component System）架构**设计，将游戏实体、属性、行为彻底解耦，告别传统面向对象继承带来的耦合困境，实现高性能数据导向的游戏逻辑。

第三，职业体系严格遵循**传奇经典三职业**（战士/法师/道士），属性体系采用**传奇经典属性面板**（HP/MP/AC/MAC/DC/MC/SC等），玩法100%还原。

**核心技术栈**：Rust + tokio + Bevy（客户端）+ ECS（服务端核心逻辑）+ egui（服务端界面）+ MySQL/Redis

### 1.2 技术栈选型理由

| 技术选型 | 选型理由 |
|---------|---------|
| **Rust**（服务端+客户端统一） | 内存安全、高性能、无GC停顿、编译期消除数据竞争；服务端和客户端共享数据结构定义代码 |
| **tokio** | Rust生态最成熟的异步运行时，服务端网络层和客户端网络层共用 |
| **Bevy** | 纯Rust游戏引擎，wgpu原生渲染，ECS架构与M2Server的ECS架构同构；社区活跃、2D支持优秀、无C++依赖 |
| **ECS架构**（M2Server） | 数据与行为分离、缓存友好、并行处理简单；游戏实体（玩家/怪物/NPC）天然适合ECS模型 |
| **egui**（服务端界面） | 纯Rust即时模式GUI，与服务端程序同进程运行；美观、现代、跨平台 |
| **Lua**（脚本引擎） | 游戏业务逻辑脚本化，热更新不停服；通过mlua与Rust集成 |
| **MySQL + Redis** | 运营团队熟悉，稳定性高，生态成熟 |
| **Protobuf**（客户端-服务端协议） | 跨语言支持好（Rust/C++）、序列化高效、版本兼容能力强 |

### 1.3 8服务架构继承自V3.0

## 第二章：Bevy客户端架构设计

### 2.1 为什么选择Bevy

传统游戏客户端开发通常选择Unity/Unreal/Cocos等引擎，但选择Bevy具有以下独特优势：

| 维度 | Unity | Unreal | Cocos | **Bevy** |
|------|-------|--------|-------|----------|
| 语言 | C# | C++ | C++/Lua | **Rust** |
| 引擎代码可读性 | 中 | 差 | 中 | **优** |
| 内存安全 | 托管GC | 手动管理 | 手动管理 | **编译期安全** |
| 服务端代码复用 | 无 | 无 | 无 | **可共享结构定义** |
| 2D游戏支持 | 优 | 一般 | **优秀** | **优** |
| 社区生态 | 成熟 | 成熟 | 成熟 | **成长中** |
| 商业案例 | 大量 | 3A游戏 | 手游为主 | **初期** |
| 学习曲线 | 低 | 高 | 中 | **中高** |

**Bevy的核心优势**：

第一，**技术栈统一**。服务端Rust代码中的数据结构（如PlayerData、ItemData、SkillData）可以直接在客户端项目中引用，无需维护两套定义，避免协议不一致问题。

第二，**ECS架构同构**。M2Server使用ECS架构处理服务端游戏逻辑，Bevy原生使用ECS架构（bevy_ecs），客户端和服务端使用相同的ECS概念，团队只需要理解一套架构。

第三，**原生wgpu渲染**。Bevy基于wgpu实现渲染，不依赖DirectX/OpenGL/Vulkan的特定绑定，天然支持多平台（Windows/Linux/macOS/Web/移动端规划中）。

第四，**开源可控**。Bevy完全开源，协议宽松（MIT/Apache-2.0），无授权费用，无商业引擎锁定风险。

### 2.3 核心模块设计

#### 2.3.1 瓦片地图渲染模块

传奇类游戏的地图由瓦片（Tilemap）构成，客户端需要高效渲染大规模瓦片地图。

**技术实现**：

- 使用 **bevy_ecs_tilemap** 插件处理瓦片渲染
- 地图数据以Chunks为单位分块加载（每个Chunk = 16x16 或 32x32 瓦片）
- 相机跟随玩家位置，只渲染可见区域内的Chunks（视锥裁剪）
- 支持多图层渲染：地面层、遮挡层、装饰层、遮挡角色层（类似传统传奇的遮挡效果）

**地图数据结构**（与服务端共享）：

```text
MapData
├── map_id: u32          // 地图ID
├── width: u32           // 地图宽度（瓦片数）
├── height: u32          // 地图高度（瓦片数）
├── tile_size: u32       // 单瓦片像素大小
├── layers: Vec<Layer>    // 地图图层
│   ├── layer_id: u8     // 图层ID（决定渲染顺序）
│   ├── tiles: Vec<u32>  // 瓦片ID数组
│   └── opacity: f32    // 图层透明度
├── walk_grid: Vec<bool> // 阻挡数据（true=可通行）
├── objects: Vec<MapObject> // 地图对象（门/花/树等装饰）
└── spawn_points: Vec<SpawnPoint> // 出生点/传送点
```

**遮挡处理**：传奇的遮挡效果通过多层瓦片和透明度实现。当角色A在角色B下方（Y轴更小）时，角色A先渲染，角色B后渲染，形成站立遮挡效果。Bevy的2D渲染顺序通过Z轴深度控制，Y轴较小的实体自动分配较小的Z值。

#### 2.3.2 网络通信模块

**技术实现**：

- 使用 **tokio** 处理异步网络IO（非Bevy内置）
- TCP连接通过 **tokio::net::TcpStream**
- WebSocket支持通过 **tokio-tungstenite**（用于Web平台）
- Protobuf序列化使用 **prost** crate

**网络层设计**：

```
NetworkManager
├── connection_state: ConnectionState    // 未连接/登录中/选服中/游戏中
├── tcp_stream: Option<TcpStream>        // TCP连接
├── send_queue: Channel<OutgoingMessage> // 发送队列
├── recv_handler: Task                   // 接收处理协程
├── heartbeat_timer: Interval            // 心跳定时器
└── reconnect_policy: ReconnectPolicy    // 重连策略

ConnectionState:
  Disconnected → LoggingIn → SelGate → InGame
```

**消息处理流程**：

1. 收到服务端消息字节流
2. 按协议格式解析（包头+长度+类型+序列号+消息体+校验）
3. 将消息反序列化为对应Protobuf结构体
4. 发送到Bevy的事件总线（EventWriter<ServerMessage>）
5. 游戏逻辑系统监听事件并响应

**断线重连**：客户端维护连接状态机，断线后自动按策略重连（3次，间隔递增）。重连成功后发送恢复会话请求，服务端返回当前游戏状态。

#### 2.3.3 玩家表现模块

**角色渲染**：

- 每个玩家实体对应一个 Bevy **Entity**
- 使用 **Sprite** 组件渲染角色精灵图
- 精灵图按职业和方向（8方向）切分
- 站立/行走/攻击/施法/死亡各有独立动画状态

**动画状态机**：

```
AnimationState
├── idle       // 站立待机
├── walk       // 行走
├── run        // 奔跑
├── attack_1   // 普通攻击
├── attack_2   // 技能攻击（不同技能不同动画）
├── spell      // 法师/道士施法
├── hurt       // 受击
├── die        // 死亡
└── sit        // 休息/打坐
```

**移动表现**：

- 客户端本地预测移动（先行平滑位移）
- 服务端校正：定期收到服务端位置同步后修正本地预测
- 平滑插值：使用 lerp（线性插值）在两帧服务端位置之间平滑过渡
- 攻击动画锁定：普攻/技能释放期间，角色朝向锁定目标

#### 2.3.4 特效与音效模块

**视觉特效**：

- 技能特效、爆炸效果使用 **Sprite Animation**（精灵序列帧动画）
- 地面光效（魔法盾、治愈光环）使用 **Sprite** 叠加在角色上
- 屏幕特效（药水效果、BUFF图标）使用 **UI层** 渲染

**音效系统**：

- 使用 **kira** 音频库（纯Rust，支持BGM/SE/3D音效）
- 背景音乐循环播放
- 音效根据事件触发（攻击命中、技能释放、怪物死亡、拾取物品）
- 3D音效根据声源与角色距离和方向计算音量衰减

### 4.3 伤害公式详解

#### 4.3.1 战士物理攻击伤害

```
普攻伤害 = random(DC下限, DC上限) - 目标AC/2 + 力量修正 + 武器幸运修正

烈火剑法 = 单次伤害 × 2 × (1 ± 5%随机波动)

逐日剑法 = (第一刀 + 第二刀 + 第三刀 + 第四刀) 总伤害
           每刀 = random(DC下限, DC上限) × 1.2
```

#### 4.3.2 法师魔法攻击伤害

```
魔法伤害 = random(MC下限, MC上限) - 目标MAC/3 + 智力修正 + 技能系数

技能系数：
  雷电术 = 1.0
  冰咆哮 = 0.6（范围分摊）
  火墙 = 0.4（持续）
  流星火雨 = 1.8（高伤害范围）
```

#### 4.3.3 道士道术攻击伤害

```
道术伤害 = random(SC下限, SC上限) - 目标MAC/3 + 精神修正 + 技能系数

技能系数：
  灵魂火符 = 0.8
  噬血术 = 1.2（额外吸血）
  召唤骷髅 = 基础攻击力 × 技能等级 × 0.3（召唤物普攻）
```

#### 4.3.4 暴击与命中

```
普攻最大伤害概率 = (幸运值 / 255) × 100%

物理命中率 = 准确值 / (准确值 + 目标敏捷值) × 基准命中率

物理闪避率 = 目标敏捷值 / (准确值 + 目标敏捷值) × 基准闪避率

暴击伤害 = 当前伤害 × 2.0（固定2倍）
```

### 4.4 防御机制

#### 4.4.1 物理防御（AC）

```
有效防御 = random(AC下限, AC上限)

实际减伤 = min(攻击伤害 × 0.5, 防御效果 × 2)

注：AC越高，减伤效果越明显，但存在最低伤害保底
```

#### 4.4.2 魔法防御（MAC）

```
有效魔御 = random(MAC下限, MAC上限)

实际魔法减伤 = min(魔法伤害 × 0.3, 魔御效果 × 3)

注：MAC对魔法伤害的减免效率低于AC对物理伤害
```

## 第五章：M2Server ECS架构设计

### 5.1 ECS架构概述

ECS（Entity Component System）是游戏开发中广泛使用的数据导向架构模式。与传统面向对象架构的对比如下：

| 维度 | 传统面向对象（V3.0） | ECS架构（V4.0） |
|------|--------------------|----------------|
| 实体表示 | 类继承树（Player/Monster继承自Entity） | Entity（简单ID） + Components（数据） |
| 属性存储 | 分散在各实体类中 | 组件数组，连续内存存储 |
| 行为实现 | 方法绑定在类中 | System（纯函数）处理组件数据 |
| 实体创建 | new Player() / new Monster() | world.spawn() + 附加Components |
| 组件添加/删除 | 继承链修改 | 动态附加/移除Components |
| 缓存友好性 | 差（对象分散，引用跳转） | 优（同类组件连续内存访问） |
| 并行处理 | 需手动加锁 | 组件系统天然并行（无共享可变状态） |
| 代码复用 | 继承层次复杂时难以复用 | System可作用于任何匹配Components的Entity |

### 5.2 M2Server ECS架构

```

│  │                                                        ││

│  └─────────────────────────────────────────────────────┘│
│                                                        │
│  ┌─────────────────────────────────────────────────────┐│
│  │                 Entity Builder                       ││
│  │  spawn_player() → 附加 PlayerTag+Position+HP+Mana... ││

│  └─────────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────────┘
                    │
                    │ 每帧Tick循环（固定频率，如每秒10帧）
                    ▼
┌──────────────────────────────────────────────────────────┐
│                    Schedule Loop                          │
│                                                        │
│  1. 接收消息队列（来自RunGate的玩家操作消息）              │
```

### 5.4 Systems（系统）设计

每个System是纯函数，接收World引用，查询匹配条件的Entities，执行数据变更。不包含任何业务数据存储。

#### MovementSystem

```text
职责：处理所有实体的移动逻辑

处理逻辑：
1. 查询所有拥有 PositionComponent + (PlayerTag | MonsterTag) 的实体
2. 对于每个实体：
   a. 若有WalkPathComponent：
      - 沿路径前进一格（根据WalkSpeed计算是否到时间）
      - 若到达路径终点，清空WalkPathComponent
   b. 若无WalkPath但有target_position：
      - 重新计算寻路路径（调用A*寻路）
   c. 若无任何移动目标：
      - 不处理
3. 处理地图边界和阻挡检测（不可通行则停止移动）
4. 触发SyncEvent（位置变化推送给客户端）

性能优化：
- 寻路计算每帧最多处理N个实体（N可配置）
- 大量实体同时寻路时排队处理
```

#### CombatSystem

```text
职责：处理所有战斗相关逻辑

处理逻辑：
1. 查询所有拥有 CombatStatsComponent + HealthComponent 的实体
2. 处理每帧到期的攻击判定：
   a. 验证目标是否有效（存在、同一地图、距离合法）
   b. 根据实体职业计算伤害（DC/MC/SC → 伤害公式）
   c. 应用目标AC/MAC防御计算
   d. 计算暴击、命中、闪避
   e. 扣除目标HP
   f. 判断目标是否死亡（HP <= 0）
3. 死亡判定后：
   a. 触发DeathSystem处理死亡逻辑
   b. 触发LootSystem处理掉落
   c. 分配经验给攻击者和队伍成员
4. 广播战斗结果（伤害数字显示、死亡效果）

关键设计：
- 战斗逻辑完全数据驱动，伤害公式通过配置表定义
- 无需修改代码即可调整数值平衡
```

#### MonsterAISystem

```text
职责：驱动怪物的AI状态机

状态机定义：
┌─────────┐
│  Idle   │ ← 待机，每帧判断是否进入巡逻或追击
└────┬────┘
     │ 检测到目标进入警戒范围
     ▼
┌─────────┐
│ Patrol  │ ← 沿巡逻路径移动，到达终点返回起点
└────┬────┘
     │ 目标进入攻击范围
     ▼
┌─────────┐
│ Chase   │ ← 向目标移动（追击）
└────┬────┘
     │ 目标离开追击范围 或 自身HP过低
     ▼
┌─────────┐
│ Return  │ ← 返回出生点
└────┬────┘
     │ 到达出生点
     ▼
┌─────────┐
│  Idle   │
└─────────┘

┌─────────┐
│ Attack  │ ← 攻击范围内目标（独立于其他状态）
└─────────┘
```

#### SkillSystem

```text
职责：处理技能释放、冷却、效果

技能释放流程（通过事件触发）：
1. 接收 SkillCastEvent（施法者Entity + 技能ID + 目标Entity/坐标）
2. 验证施法条件：
   a. 技能是否已学习
   b. MP是否足够
   c. 冷却是否结束
   d. 目标是否在有效范围内
   e. 施法者状态是否允许施法（不能死亡/眩晕中）
3. 扣除MP，启动冷却计时器
4. 触发技能效果：
   a. 即时效果：直接应用伤害/治疗
   b. 持续效果：附加BuffComponent
   c. 召唤效果：在目标位置生成PetTag实体
   d. 特殊效果：触发Lua脚本
5. 发送施法成功事件（用于客户端播放动画）
```

#### BuffSystem

```text
职责：管理所有BUFF/DEBUFF的定时效果

处理逻辑：
1. 每帧检查所有BuffListComponent
2. 对每个ActiveBuff：
   a. 递减剩余时间
   b. 若到达间隔触发时间：执行间隔效果（如周期性扣血）
   c. 若时间归零：移除Buff，触发移除效果（恢复属性等）
3. 排序BUFF优先级（某些BUFF不可叠加，取最高）

BUFF类型枚举：
  - 属性加成型（BUFF增加DC/AC/MAC等）
  - 持续伤害型（每秒扣血，如中毒）
  - 持续治疗型（每秒回血）
  - 控制型（眩晕/冰冻/沉默/定身）
  - 隐身型（客户端显示隐身）
  - 召唤型（召唤物存在期间）
```

### 5.5 ECS架构的实战优势

#### 5.5.1 性能优势示例：怪物AI并行处理

传统面向对象架构中，每个怪物是一个对象，每个对象的Update方法可能相互影响（读取/修改共享状态），必须加锁。

ECS架构下，MonsterAISystem作为单次遍历处理所有怪物：

```
// 伪代码示意（实际为Rust代码）
fn update_monster_ai(world: &mut World) {
    // 一次性读取所有怪物数据（连续内存）
    let monsters: Vec<(Entity, &Position, &AIState)> = 
        world.query::<(&Position, &AIState)>()  // Archetype查询，高效
             .with::<MonsterTag>()
             .iter()
             .collect();
    
    // 遍历所有怪物，各自独立处理（无共享可变状态）
    for (entity, position, ai_state) in monsters {
        let new_state = compute_next_state(entity, position, ai_state);
        // 修改组件数据
        world.get_mut::<AIState>(entity).unwrap().current_state = new_state;
    }
}
```

所有怪物在同一帧的处理过程中，**无任何共享可变状态**，天然支持多线程并行。只需将monsters数组分片，分配给不同线程处理，即可实现零锁并行AI计算。

#### 5.5.2 内存布局优势

ECS架构的组件按类型存储，而非按实体存储：

```
// 传统OOP（每个Entity独立内存）
Entity[0]: HP=100, AC=5,  Name="玩家A" ...
Entity[1]: HP=200, AC=10, Name="玩家B" ...  ← 不连续内存，CPU缓存miss多

// ECS（同类组件连续内存）
HP:   [100, 200, 150, 300, ...]  ← 连续内存，一次读取多个
AC:   [5,   10,  7,   15,   ...]  ← 连续内存
Name: ["玩家A", "玩家B", ...]     ← 字符串单独区域
```

当CombatSystem处理AC（防御）时，只需访问AC组件的连续内存块，CPU缓存命中率极高，性能提升显著。

#### 5.5.3 实体类型扩展优势

假设需要新增"宠物"实体类型，传统OOP需要新建Pet类继承Entity，或修改基类。ECS架构只需：

```rust
// 创建宠物实体 = 选择性附加组件
let pet = world.spawn();
world.insert(pet, (
    PlayerInfoComponent { /* ... */ },
    HealthComponent { hp: 500, max_hp: 500, regen_hp: 0 },
    CombatStatsComponent { dc_min: 10, dc_max: 20, /* ... */ },
    PositionComponent { x: 100, y: 200, /* ... */ },
    MonsterTag,  // 复用MonsterTag使AI系统也驱动宠物
    PetTag,      // 额外标识这是宠物
    WalkPathComponent { /* 宠物跟随主人的路径 */ },
));
```

无需修改任何现有System，Pet就具备了所有需要的行为（移动、战斗、AI）。这就是ECS的**组合优于继承**原则。

### 5.6 ECS与Lua脚本的集成

Lua脚本用于处理无法预定义的复杂业务逻辑（如任务流程、特殊活动、NPC对话分支），但脚本只调用Rust暴露的API，不直接操作ECS。

```
┌─────────────────────────────────────────────────────────────┐
│  Lua脚本（处理复杂业务逻辑）                                   │

```

```
│   ├── health.rs
```

**共享crate的优势**：

- 协议定义只写一次，服务端和客户端保持100%一致
- 伤害公式代码在服务端和客户端各有一份（客户端用于本地演示，服务端为权威）
- 物品配置、怪物配置、地图配置等常量共享定义
- 修改配置时只需改一处，编译时两端同时生效

## 第七章：服务间通信协议（V4.0更新）

### 7.1 协议序列化选型：Protobuf

V4.0将服务间通信和客户端-服务端通信统一使用 **Protocol Buffers**（通过prost crate）。

**选择理由**：

| 对比项 | 自研二进制 | Protobuf | FlatBuffers | Cap'n Proto |
|-------|-----------|----------|-------------|-------------|
| 跨语言 | 需手写编解码 | 原生支持Rust/C++/Java/Go等 | 原生支持 | 原生支持 |
| 工具链 | 无 | 成熟（protoc编译器）| 成熟 | 成熟 |
| 性能 | 最优（手写） | 优 | 极高（零拷贝）| 极高 |
| 版本兼容 | 需手写兼容 | 自动兼容 | 自动兼容 | 自动兼容 |
| 学习成本 | 高（需设计格式）| 低 | 中 | 中 |
| 代码生成 | 无 | 自动生成Rust/C++代码 | 自动生成 | 自动生成 |

### 10.3 核心设计立场

**立场一：ECS是M2Server的必然选择。** 传奇类MMO的核心挑战是大规模实体（数千玩家+怪物+NPC）的并发处理。ECS的数据局部性、缓存友好性、无锁并行能力，是应对这一挑战的最佳架构。面向对象架构在5000人规模下将面临严重的性能瓶颈和并发锁竞争。

**立场二：Bevy是Rust游戏客户端的最优解。** 虽然Bevy的社区不如Unity成熟，但其纯Rust技术栈、ECS同构性、无GC性能、无商业锁定等优势，使其成为本项目的最佳选择。客户端的渲染质量和稳定性（2D游戏场景下）完全可满足传奇类游戏需求。

**立场三：技术栈统一是长期投资。** 服务端Rust + 客户端Bevy/Rust + Protobuf共享定义，意味着团队只需要精通一门语言和一个生态。虽然学习曲线较单一Unity方案陡峭，但长期看代码复用率高、技术债务少、团队技能可迁移。

**立场四：经典传奇玩法不可妥协。** 三职业体系（战/法/道）、经典属性（DC/MC/SC/AC/MAC）、经典伤害公式、经典掉落体验——这些是传奇玩家的核心记忆。现代化技术只是手段，玩法体验才是目的。

**立场五：8服务架构保持不变。** 传统8服务架构（LoginGate/SelGate/RunGate/LoginSrv/DBServer/M2Server/LogServer/GameCenter）经二十年运营验证，运维人员熟悉，改动风险高收益低。只改变实现技术，不改变服务划分。

### 10.4 风险提示

| 风险 | 等级 | 应对 |
|-----|------|------|
| Bevy在大规模MMO客户端中缺少商业案例 | 中 | 初期注重核心功能，渲染方案可在后期替换为更成熟的wgpu直接调用 |
| ECS架构对团队有一定学习成本 | 低 | 安排ECS专项培训，bevy_ecs文档完善 |
| Protobuf对实时性要求极高的场景有轻微序列化开销 | 低 | Protobuf序列化/反序列化开销在1ms以内，对游戏逻辑帧（100ms）可忽略 |
| Rust编译时间较长影响开发效率 | 低 | 使用增量编译，合理划分crate，CI/CD预编译 |

## 第十一章：安全与反外挂体系

### 11.1 安全设计原则

游戏服务端安全的核心原则是**"客户端数据不可信"**。所有游戏逻辑（伤害计算、物品归属、位置移动）必须在服务端独立验证，客户端只负责表现和输入采集。

### 11.2 服务端校验策略

#### 11.2.1 移动校验（防瞬移）

| 校验项 | 校验规则 | 处理方式 |
|--------|---------|---------|
| 移动频率 | 两次移动请求间隔不得低于移动一格所需最小时间 | 超出阈值则忽略该次移动 |
| 移动距离 | 单次移动请求的起点坐标必须与最近一次服务端确认坐标一致 | 不一致则拉回上一次确认坐标 |
| 路线合理性 | 若单次移动超过1格，需验证中间路径无阻挡 | 路径有阻挡则拉回 |
| 地图边界 | 目标坐标必须在地图合法范围内 | 出界则忽略 |

**服务端位置状态机**：

```
客户端发送移动请求(FromX, FromY, ToX, ToY)
         │
         ▼
  服务端校验FromX,FromY是否为已确认坐标
         │
    ┌────┴────┐
    │  匹配    │  不匹配
    ▼         ▼
 执行移动    拉回到已确认坐标
         │
         ▼
  服务端发送位置同步给客户端
```

#### 11.2.2 攻击频率校验（防加速器）

| 校验项 | 校验规则 | 处理方式 |
|--------|---------|---------|
| 攻击间隔 | 两次攻击的时间间隔不得低于当前攻速对应的最小间隔 | 低于阈值则忽略 |
| 技能冷却 | 技能释放时校验冷却是否到期 | 冷却未到则拒绝 |
| MP消耗 | 攻击/技能消耗MP前，验证服务端记录的MP值是否足够 | MP不足则拒绝 |
| 攻击距离 | 攻击目标必须在武器/技能有效范围内 | 超出范围则判定为无效攻击 |

#### 11.2.3 物品操作校验（防复制）

| 校验项 | 校验规则 | 处理方式 |
|--------|---------|---------|
| 物品归属 | 物品操作（丢弃/交易/使用）时，服务端验证物品是否存在于玩家背包 | 不存在则拒绝 |
| 数量校验 | 物品数量操作前，验证服务端记录的数量 | 数量不足则拒绝 |
| 交易双向锁定 | 交易过程中，双方背包锁定，一方取消则全部回滚 | 数据库事务保证原子性 |
| 拾取归属 | 掉落物品的拾取采用归属判定（归属权过期后才可他人拾取） | 服务端验证归属权 |

#### 11.2.4 战斗数值校验（防数据篡改）

| 校验项 | 校验规则 | 处理方式 |
|--------|---------|---------|
| 伤害合理性 | 伤害值不得超过服务端根据公式计算的最大值 | 超出则按服务端计算值处理 |
| 属性上限 | 装备/道具增加的属性不得超过配置上限 | 超出则截断 |
| HP/MP合理性 | 客户端上报的HP/MP不得高于服务端记录的最大值 | 差异过大触发异常告警 |

### 11.3 异常行为检测

#### 11.3.1 行为异常检测指标

| 检测项 | 检测逻辑 | 阈值 | 处理方式 |
|--------|---------|------|---------|
| 秒怪检测 | 单个玩家单位时间内击杀怪物数量 | 超阈值记录，累积触发 | 记录日志，轻度警告 |
| 异常移动 | 短时间内移动距离异常大 | 超出单帧最大移动距离 | 记录日志并触发人机验证 |
| 高频交易 | 同一玩家高频发起交易 | 超出正常交易频率 | 临时限制交易功能 |
| 脚本机器人 | 操作间隔极度规律（无随机性） | 统计操作间隔方差极低 | 标记为疑似机器人 |
| 异常资源获取 | 单位时间内资源（金币/物品）获取量异常 | 超阈值触发审计 | 冻结异常物品，数据回滚 |

#### 11.3.2 封号机制

| 封号等级 | 触发条件 | 封号时长 | 处理方式 |
|---------|---------|---------|---------|
| 警告 | 首次轻微异常 | — | 发送警告消息，记录档案 |
| 临时封禁 | 轻微违规累积或单次中等异常 | 1-7天 | 禁止登录，游戏数据冻结 |
| 永久封禁 | 明确使用外挂或严重数据篡改 | 永久 | 禁止登录，标记账号，禁止解封 |

**封号执行流程**：GM在GameCenter发起封号 → 写入账号封禁表 → LoginSrv实时查询 → 下次登录拒绝 → 记录日志（LogServer）。

### 11.4 数据传输加密

| 层级 | 加密方案 | 密钥管理 | 用途 |
|------|---------|---------|------|
| 登录阶段 | RSA非对称加密 | 服务端公钥预置客户端 | 交换会话密钥 |
| 游戏阶段 | AES-128-CFB对称加密 | RSA交换后建立 | 所有游戏数据传输加密 |
| Token | JWT令牌 | 服务端维护，Redis存储 | 登录令牌、进入令牌 |
| 数据库 | 密码加盐哈希 | Argon2id | 账号密码存储 |

## 第十二章：Lua热更新方案

### 12.1 热更新设计目标

Lua脚本热更新的核心目标是：**在不重启M2Server进程、不中断玩家游戏的情况下，更新游戏业务逻辑。**

### 12.2 热更新范围划分

| 类型 | 可热更新 | 需要重启 | 说明 |
|------|:--------:|:--------:|------|
| NPC对话脚本 | ✅ | — | NPC点击后的对话分支逻辑 |
| 任务脚本 | ✅ | — | 任务接受/完成/奖励逻辑 |
| 活动脚本 | ✅ | — | 定时活动开启/关闭逻辑 |
| 物品使用效果脚本 | ✅ | — | 特殊物品使用后的效果 |
| 技能效果脚本 | ✅ | — | 技能的额外效果（部分可热更） |
| 伤害公式 | — | ✅ | 基础伤害计算公式 |
| 掉落概率表 | ✅ | — | 怪物掉落配置 |
|怪物AI行为树 | ✅ | — | 怪物状态机行为配置 |
| 配置表（物品/技能/怪物）| ✅ | — | 配置数值类数据 |
| ECS Systems代码 | — | ✅ | Rust代码无法热更新 |
| Protobuf协议定义 | — | ✅ | 协议变更需要重启 |

### 12.3 Lua脚本热更新机制

#### 12.3.1 脚本加载架构

```
M2Server进程
│
├── 脚本运行时（mlua）← 每个脚本类型一个独立LuaState
│   ├── quest_lua_state    → 任务脚本
│   ├── npc_lua_state      → NPC对话脚本
│   ├── activity_lua_state → 活动脚本
│   └── item_lua_state     → 物品效果脚本
│
├── 脚本文件系统（监控）
│   └── /scripts/
│       ├── quests/        → 任务脚本目录
│       ├── npc/           → NPC脚本目录
│       ├── activities/    → 活动脚本目录
│       └── items/         → 物品脚本目录
│
└── 脚本API层（Rust暴露给Lua）
    └── 提供give_item / take_item / teleport / give_exp等API
```

#### 12.3.2 热更新流程

```
GM在GameCenter点击"热更新脚本"
           │
           ▼
   GameCenter发送热更新命令到M2Server
           │
           ▼
   M2Server停止触发该类脚本的新调用（等待现有调用完成）
           │
           ▼
   LuaState.close() 关闭旧Lua运行时
           │
           ▼
   重新加载该目录下的所有.lua文件到新LuaState
           │
           ▼
   验证新脚本语法正确（编译测试）
           │
      ┌────┴────┐
      │  成功    │  失败
      ▼         ▼
  切换为新LuaState   回滚为旧LuaState
  通知GameCenter成功   通知GameCenter失败原因
```

#### 12.3.3 脚本变更检测

| 检测方式 | 说明 | 适用场景 |
|---------|------|---------|
| 文件MD5对比 | 监控脚本文件MD5值，变化则触发更新 | 生产环境推荐 |
| 定时扫描 | 每N分钟扫描脚本目录，发现变化则触发 | 开发环境 |
| 手动触发 | GM手动点击热更新 | 紧急修复 |
| Git Hook触发 | CI/CD在脚本文件提交后自动触发 | 有CI/CD流水线的团队 |

### 12.4 配置表热更新

配置表（物品属性、怪物属性、技能属性）以TOML/JSON格式存储于配置文件目录，通过GameCenter可触发配置重载。

**配置热更新流程**：

```
GM在GameCenter修改配置表 → 保存到配置文件 → 点击"重载配置"
           │
           ▼
    M2Server读取新配置文件
           │
           ▼
    验证配置格式正确（字段完整、类型正确、数值合理）
           │
      ┌────┴────┐
      │  成功    │  失败
      ▼         ▼
  更新内存中的配置缓存   保留旧配置，报告错误
  新玩家操作使用新配置
  正在战斗的实体继续使用旧配置（避免战斗中属性突变）
           │
           ▼
    下一次进入战斗时切换到新配置
```

## 第十三章：协议版本兼容与客户端管理

### 13.1 版本号管理体系

| 版本类型 | 定义 | 示例 | 管理策略 |
|---------|------|------|---------|
| **协议版本** | 客户端与服务端通信协议格式的版本 | 1.0.0 | 严格向前兼容 |
| **客户端版本** | 客户端程序完整版本号 | 1.2.5 | 支持多个版本同时在线 |
| **服务端版本** | M2Server等后端服务的版本号 | 2.1.0 | 对客户端版本有最低要求 |

### 13.2 协议版本控制机制

**Protobuf字段编号策略**：Protobuf通过字段编号（而非字段名称）标识字段，编号一旦确定不可更改，只能新增编号。删除字段时保留编号（标注deprecated）。

```
// 协议v1.0.0
message PlayerData {
  uint64 char_id = 1;
  string name = 2;
  uint32 level = 3;
  // 字段4已删除，原字段保留编号但不使用
  repeated SkillRecord skills = 5;
}

// 协议v1.1.0（新增字段，字段编号连续）
message PlayerData {
  uint64 char_id = 1;
  string name = 2;
  uint32 level = 3;
  // deprecated字段保留编号
  // field 4 已废弃
  repeated SkillRecord skills = 5;
  uint32 pk_value = 6;        // 新增：PK值
  GuildBrief guild = 7;        // 新增：行会信息
}

// 协议v2.0.0（Breaking Change，需要升级客户端）
// 不兼容变更需要告知客户端团队发布新版本
```

### 13.3 多客户端版本兼容策略

```
客户端请求登录（携带客户端版本号）
           │
           ▼
  LoginSrv查询客户端版本兼容性配置
           │
     ┌─────┼─────┐
     │     │     │
  支持   不支持  强制更新
  但旧版   │
     │     ▼
     ▼   LoginSrv返回"需要更新客户端"
   正常   （客户端跳转至更新下载页面）
  登录
```

**兼容性矩阵配置**：

| 客户端版本 | 服务端最低支持版本 | 功能限制 | 推荐升级 |
|-----------|-----------------|---------|---------|
| 1.0.x | 1.0.x | 旧协议，无新功能 | ✅ 建议升级 |
| 1.1.x | 1.0.x | 无限制 | 建议升级 |
| 1.2.x | 1.0.x | 无限制 | 当前最新 |
| 2.0.x | 2.0.x | — | 新版特性 |

### 13.4 强制更新机制

**触发条件**：
- 服务端检测到客户端版本低于最低兼容版本
- 服务端新增了Breaking Change协议
- 安全漏洞需要客户端升级

**强制更新流程**：
1. 服务端返回升级提示（附带最新版本号和下载地址）
2. 客户端弹出升级提示，禁止进入游戏
3. 玩家下载并安装最新版本
4. 新版本客户端重新登录

## 第十四章：装备强化与炼器系统

### 14.1 系统概述

装备强化是传奇类游戏的核心氪金系统之一，V4.0实现完整的炼器体系：

| 子系统 | 核心功能 |
|-------|---------|
| **强化系统** | 消耗强化石和金币，提升装备强化等级（+1至+15） |
| **继承系统** | 将旧装备的强化等级无损转移到新装备上 |
| **分解系统** | 将不需要的装备分解为材料 |
| **追加系统** | 装备可追加随机属性，属性条数和品质分级 |
| **鉴定系统** | 未鉴定装备可通过鉴定书获得随机属性 |

### 14.2 强化系统设计

#### 14.2.1 强化规则

| 强化等级 | 成功率 | 消耗材料 | 失败惩罚 |
|---------|--------|---------|---------|
| +1→+2 | 100% | 强化石×1 + 金币 | — |
| +2→+3 | 90% | 强化石×2 + 金币 | — |
| +3→+4 | 80% | 强化石×3 + 金币 | 强化等级-1 |
| +4→+5 | 70% | 强化石×4 + 金币 | 强化等级-1 |
| +5→+6 | 60% | 强化石×5 + 金币 | 强化等级归零 |
| +6→+7 | 50% | 强化石×8 + 金币 | 装备消失 |
| +7→+8 | 40% | 强化石×10 + 金币 | 装备消失 |
| +8→+9 | 30% | 强化石×15 + 金币 | 装备消失 |
| +9→+10 | 20% | 强化石×20 + 金币 | 装备消失 |
| +10→+11 | 15% | 强化石×30 + 金币 | 装备消失 |
| +11→+12 | 10% | 强化石×40 + 金币 | 装备消失 |
| +12→+13 | 5% | 强化石×50 + 金币 | 装备消失 |
| +13→+14 | 3% | 强化石×80 + 金币 | 装备消失 |
| +14→+15 | 1% | 强化石×100 + 金币 | 装备消失 |

**强化冷却时间**：每次强化后需等待10秒才能再次强化同一装备。

#### 14.2.2 强化属性加成

| 强化等级 | 额外属性加成 |
|---------|------------|
| +1~+3 | DC/MC/SC +1/级 |
| +4~+6 | DC/MC/SC +2/级，AC/MAC +1/级 |
| +7~+10 | DC/MC/SC +3/级，AC/MAC +2/级，HP +50/级 |
| +11~+15 | DC/MC/SC +5/级，AC/MAC +3/级，HP +100/级 |

### 14.3 继承系统设计

**继承规则**：
- 目标装备必须为未强化或强化等级低于原材料装备
- 继承后，原材料装备强化等级归零，原材料保留
- 继承不消耗金币，只消耗继承石
- 武器与武器之间继承，衣服与衣服之间继承（部位限制）

### 14.4 分解系统设计

**分解规则**：
- 白色普通装备：分解为少量金币
- 蓝色稀有装备：分解为强化石碎片
- 紫色精英装备：分解为强化石
- 橙色史诗装备：分解为高纯度强化石

### 14.5 ECS中的装备系统实现

装备强化/继承/分解等操作，在ECS中通过InventorySystem处理：

```
装备强化流程（ECS事件驱动）：
1. 玩家点击强化 → 客户端发送强化请求（装备UID、目标等级）
2. RunGate转发至M2Server
3. M2Server事件：EnhanceEvent(char_id, equip_uid, target_level)
4. InventorySystem处理：
   a. 验证装备存在且属于玩家
   b. 验证玩家拥有足够的强化石和金币
   c. 查询强化配置表获取成功率
   d. 随机判定成功/失败
   e. 成功：更新装备强化等级Component，扣除材料
   f. 失败：按失败惩罚扣减强化等级或删除装备
   g. 保存更新至DBServer
5. 广播结果给客户端（强化成功/失败动画和消息）
```

### 15.3 攻城战详细流程

#### 15.3.1 宣战期

- 攻城行会在沙巴克管理员NPC处缴纳攻城押金（金币或道具）
- 攻城行会名称显示在沙巴克地图入口
- 所有行会成员可进入沙巴克地图（地图内进入PK模式）
- 沙巴克当前占领行会成员获得临时属性加成（守方优势）

#### 15.3.2 战争期

**占领旗帜机制**：

| 阶段 | 占领状态 | 规则 |
|------|---------|------|
| 开战 | 无旗帜 | 地图中央存在旗帜 |
| 第一面旗 | 攻方占领 | 攻方行会成员触碰旗帜后开始倒计时（5分钟） |
| 坚守 | 攻方持续占领 | 倒计时归零 → 攻方占领第一面旗 |
| 第二面旗 | 攻方占领 | 同上，占领后守方需要重新攻占 |
| 夺回 | 守方占领 | 守方触碰己方旗帜可重新占领 |

**占领判定规则**：
- 旗帜区域内有且仅有某一方的行会成员，则该方开始占领倒计时
- 区域内双方都有成员，占领暂停
- 倒计时归零，旗帜变色（攻方颜色），该行会获得占领权

**战争期结束判定**：
- 60分钟结束时，占领旗帜数量多的行会获胜
- 若时间结束时无任何旗帜被占领，守方（沙巴克原占领行会）获胜
- 若无行会攻城，守方维持占领

#### 15.3.3 战后奖励

| 奖励项 | 占领行会 | 说明 |
|-------|---------|------|
| 行会资金 | ✅ | 每日可领取行会资金（沙巴克税收） |
| 专属称号 | ✅会长 | "沙巴克城主"称号 |
| 进入权限 | ✅行会成员 | 沙巴克专属地图（每日进入次数） |
| 攻城记录 | ✅ | 城主雕像矗立于沙巴克城内 |
| 战败行会 | — | 无奖励，可参加下一次攻城 |

### 15.4 ECS中沙巴克系统实现

沙巴克作为特殊的活动系统，通过ScheduleSystem定时任务驱动：

```
沙巴克ECS组件（扩展）：
  SaccoonComponent
    - current_state: SaccoonState  // NonWar/PreWar/War
    - occupy_guild_id: Option<u64> // 当前占领行会
    - flag_states: [FlagState; 3]  // 三面旗帜状态
    - attack_guild_id: Option<u64> // 攻城行会
    - war_start_time: u64          // 战争开始时间戳
    - flag_capture_timers: [u64; 3] // 旗帜占领倒计时

ScheduleSystem处理：
  - 每周六 20:00：触发 SaccoonStartEvent → PreWar状态
  - 每周六 20:30：触发 SaccoonWarStartEvent → War状态，刷新怪物
  - 每周六 21:30：触发 SaccoonWarEndEvent → 结算胜负 → NonWar状态
```

## 第十六章：数据回滚方案

### 16.1 回滚触发场景

| 触发场景 | 回滚类型 | 说明 |
|---------|---------|------|
| GM误操作 | 手动回滚 | GM错误地修改了玩家数据，主动回滚 |
| 异常数据 | 自动告警回滚 | 系统检测到数据异常，自动回滚 |
| 玩家申诉 | 审批回滚 | 玩家申诉装备丢失/数据异常，审批后回滚 |
| 系统故障 | 批量回滚 | 系统故障导致数据错误，批量回滚受影响玩家 |

### 16.2 回滚粒度

| 粒度 | 适用场景 | 回滚内容 |
|------|---------|---------|
| **单角色数据** | 玩家装备/金币异常 | 该角色的所有数据进行回滚 |
| **单表回滚** | 单类数据错误 | 如仅装备数据异常，只需回滚装备表 |
| **单字段回滚** | 单个属性错误 | 如仅金币数值错误，回滚金币字段 |
| **指定时间点回滚** | 精确恢复 | 回滚到故障发生前N分钟的状态 |

### 16.3 数据快照策略

| 快照类型 | 频率 | 保留周期 | 存储位置 |
|---------|------|---------|---------|
| 全量快照 | 每日凌晨2点 | 30天 | 本地磁盘 + 异地对象存储 |
| 增量快照 | 每小时 | 7天 | 本地磁盘 |
| 实时Binlog | 持续 | 7天 | 本地磁盘（MySQL Binlog） |
| 操作日志快照 | 关键操作时 | 永久 | LogServer数据库 |

### 16.4 回滚执行流程

```
GM发起回滚请求（GameCenter界面）
           │
           ▼
  选择回滚玩家 + 回滚时间点/快照
           │
           ▼
  从最近快照恢复该玩家的数据到临时区域
           │
           ▼
  对比临时区域数据与当前数据的差异
           │
           ▼
  GM预览差异内容，确认回滚范围
           │
      ┌────┴────┐
      │  确认   │  取消
      ▼         ▼
  执行回滚   取消操作
  (在事务中：
  1. 锁定玩家账号
  2. 用快照数据覆盖当前数据
  3. 记录回滚日志
  4. 解锁账号)
           │
           ▼
  玩家下次登录或主动刷新时加载新数据
           │
           ▼
  发送通知给GM（回滚完成）
```

### 16.5 数据完整性保障

- **回滚前备份**：每次回滚操作前，自动将当前数据备份为"预回滚快照"，防止回滚错误
- **事务保证**：回滚操作使用MySQL事务，失败自动回滚
- **操作审计**：所有回滚操作记录到GM操作日志（LogServer），包含操作人、操作时间、回滚原因、回滚前后数据对比
- **通知机制**：回滚完成后，通知相关GM和玩家（若有）

## 第十七章：多区合服方案

### 17.1 合服触发条件

| 触发条件 | 说明 |
|---------|------|
| 服务器活跃玩家数持续低于阈值 | 如连续30天在线峰值低于200人 |
| 运营决策 | 多个老区合并集中运营 |
| 服务器硬件维护 | 物理迁移合并 |

### 17.2 合服冲突处理规则

#### 17.2.1 账号冲突处理

| 场景 | 处理规则 | 实现方式 |
|------|---------|---------|
| 同一平台账号多区存在 | 合并为一个账号，角色共存 | 账号ID统一，角色表合并 |
| 角色名重复 | 保留先创建的角色，后创建的需强制改名 | 合服后首次登录时强制改名界面 |
| 行会名重复 | 保留先创建的行会，后创建的需强制改名 | 合服后行会会长收到强制改名通知 |

#### 17.2.2 数据合并规则

| 数据类型 | 合服处理规则 |
|---------|------------|
| 角色数据 | 各区角色数据全部保留，合并到同一账号下 |
| 行会数据 | 各区行会数据全部保留，成员列表合并 |
| 物品/金币 | 各区角色物品/金币各自独立保留，不跨区合并 |
| 排行榜数据 | 全部清空，合服后根据合服后数据重新计算 |
| 沙巴克占领 | 清空，合服后首周无沙巴克攻城 |

#### 17.2.3 服务器标识

合服后：
- 保留所有区服的入口地址（旧地址可重定向到新区服列表）
- 玩家数据中标记来源服务器ID（用于区分原服务器）
- 合服公告通知所有玩家

### 17.3 合服执行流程

```
合服前准备（提前3天公告）
           │
           ▼
  备份所有服务器数据（全量快照）
           │
           ▼
  停止所有服务器的登录（新玩家禁止进入）
           │
           ▼
  等待所有老玩家自然下线（最长等待24小时）
           │
           ▼
  导出所有区的数据
           │
           ▼
  数据清洗与合并
  (账号去重/角色名冲突处理/行会名冲突处理)
           │
           ▼
  导入合并后的数据到新区服数据库
           │
           ▼
  验证数据完整性（账号数/角色数/物品数对账）
           │
           ▼
  启动新区服，开放登录
           │
           ▼
  合服公告 + 补偿发放（合服礼包）
```

## 第十八章：支付与商城系统

### 18.1 支付系统架构

支付系统采用独立模块设计，与游戏核心逻辑解耦：

```
┌──────────────────────────────────────────────────────────────┐
│                      客户端商城界面                             │
│         （Bevy UI / Web商城页面）                              │
└────────────────────────────┬─────────────────────────────────┘
                             │ 商品列表/订单请求
                             ▼
┌──────────────────────────────────────────────────────────────┐
│                     PayServer（支付服务）                      │
│                    Rust + tokio + egui                       │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  订单管理：生成订单/查询订单/关闭订单                     │   │
│  │  支付渠道对接：支付宝/微信/Apple IAP/Google Play         │   │
│  │  发货管理：验证支付结果/发放元宝/记录日志                  │   │
│  │  对账管理：日对账/月对账/渠道差异处理                      │   │
│  └──────────────────────────────────────────────────────┘   │
└────────────────────────────┬─────────────────────────────────┘
                             │ 订单验证/发货指令
                             ▼
┌──────────────────────────────────────────────────────────────┐
│                       游戏数据库                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ PayOrder表  │  │ Account表   │  │ Character表          │  │
│  │ (订单记录)   │  │ (账号元宝)  │  │ (角色元宝/物品)       │  │
│  └─────────────┘  └─────────────┘  └─────────────────────┘  │
└──────────────────────────────────────────────────────────────┘
```

### 18.2 支付流程

```
1. 玩家在商城选择商品（确定商品ID和价格）
2. 客户端向PayServer发起下单请求（商品ID、char_id）
3. PayServer生成订单（OrderID），记录到PayOrder表，状态=待支付
4. PayServer返回订单信息（OrderID、支付地址/二维码）
5. 客户端展示支付二维码/跳转支付页面
6. 玩家完成支付（支付宝/微信/App Store）
7. 支付渠道通知PayServer（异步回调，带OrderID和支付状态）
8. PayServer验证签名，确认支付成功
9. PayServer更新Order状态=已支付，并发放元宝：
   a. 查询Account表，增加元宝余额
   b. 或直接发放物品到Character背包
10. PayServer记录发货日志（LogServer）
11. 通知客户端支付成功
```

### 18.3 支付安全

| 安全措施 | 说明 |
|---------|------|
| 回调签名验证 | 所有支付渠道回调必须携带签名，PayServer验证签名后才处理 |
| 订单状态机 | 订单状态严格按流程流转（待支付→已支付→已发货），不可逆 |
| 幂等性保证 | 同一OrderID的回调只处理一次，防止重复发货 |
| 金额校验 | 回调金额必须与订单金额一致，不一致则拒绝 |
| 日志审计 | 所有支付操作记录详细日志（订单号/金额/玩家/时间/渠道） |

### 18.4 商品类型

| 商品类型 | 说明 | 示例 |
|---------|------|------|
| 直购元宝 | 购买固定数量元宝 | 60元宝、300元宝、980元宝 |
| 月卡/季卡 | 时限性权益 | 月卡（30天）、季卡（90天） |
| 礼包 | 固定组合商品 | 新手礼包、冲级礼包 |
| 限时折扣 | 限时打折商品 | 周末8折 |
| 单品直购 | 直接购买游戏物品 | 特殊坐骑、限定时装 |
| 月卡每日礼包 | 每日领取 | 月卡用户每日可领取一次 |

### 19.3 最终立场声明

**立场一：ECS是M2Server的必然选择。** 传奇类MMO的核心挑战是大规模实体（数千玩家+怪物+NPC）的并发处理。ECS的数据局部性、缓存友好性、无锁并行能力，是应对这一挑战的最佳架构。面向对象架构在5000人规模下将面临严重的性能瓶颈和并发锁竞争。

**立场二：Bevy是Rust游戏客户端的最优解。** 虽然Bevy的社区不如Unity成熟，但其纯Rust技术栈、ECS同构性、无GC性能、无商业锁定等优势，使其成为本项目的最佳选择。客户端的渲染质量和稳定性（2D游戏场景下）完全可满足传奇类游戏需求。

**立场三：技术栈统一是长期投资。** 服务端Rust + 客户端Bevy/Rust + Protobuf共享定义，意味着团队只需要精通一门语言和一个生态。虽然学习曲线较单一Unity方案陡峭，但长期看代码复用率高、技术债务少、团队技能可迁移。

**立场四：经典传奇玩法不可妥协。** 三职业体系（战/法/道）、经典属性（DC/MC/SC/AC/MAC）、经典伤害公式、经典掉落体验、强化继承分解——这些是传奇玩家的核心记忆。现代化技术只是手段，玩法体验才是目的。

**立场五：8服务架构保持不变。** 传统8服务架构（LoginGate/SelGate/RunGate/LoginSrv/DBServer/M2Server/LogServer/GameCenter）经二十年运营验证，运维人员熟悉，改动风险高收益低。只改变实现技术，不改变服务划分。

**立场六：安全是生命线。** 服务端权威校验是反外挂的核心，所有客户端数据在服务端必须重新验证，任何绕过服务端校验的行为都必须被检测和阻止。

**立场七：运营工具决定运营效率。** GameCenter、GM系统、数据回滚、支付对账——这些运营辅助系统必须与核心游戏逻辑同等重视，它们是长期稳定运营的基础保障。

### 19.4 最终验收标准

| 验收项 | 目标值 |
|-------|-------|
| 单M2Server承载在线人数 | ≥ 5000人 |
| 单玩家消息处理延迟 | < 20毫秒 |
| 服务稳定运行时长 | 连续7天无崩溃 |
| 内存泄漏率 | 24小时运行内存增量 < 50MB |
| Lua热更新恢复时间 | < 5秒（不影响在线玩家） |
| 核心功能完成度 | 100%（所有系统全部实现） |
| GM工具可用性 | 100%（所有GM指令可执行） |
| 数据安全 | 所有关键操作有审计日志，所有数据可恢复至任意时间点 |
| 支付安全 | 零掉单、零重复发货 |
| 协议兼容性 | 支持任意客户端版本平滑接入 |
