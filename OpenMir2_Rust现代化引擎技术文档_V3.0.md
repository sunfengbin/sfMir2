# OpenMir2 Rust现代化引擎 技术文档 V3.0

**文档编号**：OM2-TD-2024-V3.0
**版本状态**：归档版
**编制目的**：基于传统8服务架构的Rust现代化引擎技术方案
**适用范围**：经典MMORPG引擎架构参考与实现指南

---

## 文档概述

本文档定义的引擎架构严格遵循传统Mir2/传奇引擎的**8服务经典架构**，保留运营人员熟悉的管理界面和使用习惯，同时将底层实现全面现代化。每个服务均为**独立可执行文件**，具备**独立egui管理界面**，服务间通过内部协议通信。

**核心设计理念**：外形如传统引擎，内部全现代化。运营人员看到的依然是熟悉的登录网关、选服网关、运行网关和各类服务器窗口，但底层实现采用Rust异步IO、无GC内存管理、现代协议设计。

## 第一章：传统架构溯源与现代化映射

### 1.1 传统传奇引擎8服务架构

经典Mir2引擎采用网关层与业务服务层分离的设计。该架构经过二十年运营验证，稳定性极高，运维人员熟悉度好。

**网关层（3个）** 负责客户端网络连接的接入，不处理业务逻辑，仅做协议转发和连接管理。

| 网关名称 | 经典端口 | 核心职责 |
|---------|---------|---------|
| **LoginGate** | 7000 | 玩家登录入口，接收客户端登录请求，转发至LoginSrv |
| **SelGate** | 7100 | 角色选择入口，处理角色列表查询、创建角色 |
| **RunGate** | 7200 | 游戏运行入口，游戏内所有消息的转发枢纽 |

**业务服务层（5个）** 处理各类业务逻辑，不直接与客户端通信，仅与网关或其他业务服务通信。

| 服务名称 | 经典端口 | 核心职责 |
|---------|---------|---------|
| **LoginSrv** | 5000 | 账号认证服务，处理账号密码验证、登录验证码生成 |
| **DBServer** | 6000 | 数据库服务，所有数据读写的唯一入口，提供数据缓存 |
| **M2Server** | 7400 | 主引擎服务，核心游戏逻辑：地图/战斗/NPC/怪物/技能/物品 |
| **LogServer** | 8000 | 日志服务，集中收集所有服务的运行日志和玩家行为日志 |
| **GameCenter** | 9000 | 游戏管理中心，统一管理和监控所有其他服务 |

### 1.2 架构拓扑关系

```
                      ┌──────────────┐
                      │  GameCenter  │
                      │  (管理中心)   │
                      └──────┬───────┘
                             │ 管理/监控
        ┌────────────────────┼─────────────────────┐
        ▼                    ▼                    ▼
   ┌───────────┐      ┌───────────┐        ┌───────────┐
   │ LoginSrv  │      │ DBServer  │        │ LogServer │
   │ (登录服务) │      │ (数据库服务) │        │ (日志服务) │
   └────┬──────┘      └────┬──────┘        └────┬──────┘
        │                  │                    │
        │ 验证请求         │ 数据读写            │ 日志写入
        ▼                  ▼                    ▼
   ┌───────────┐      ┌───────────┐        ┌───────────┐
   │ LoginGate │      │  SelGate  │◀───┐   │  RunGate  │
   │ (登录网关) │      │ (选服网关) │    │   │ (运行网关) │
   └────┬──────┘      └────┬──────┘    │   └────┬──────┘
        │                  │           │        │
```

### 1.3 现代化映射方案

每个传统服务映射为一个独立的Rust可执行程序，保持相同的职责范围和对外接口风格，内部实现全面现代化。

| 传统组件 | 现代化实现 | 内部技术栈 |
|---------|-----------|-----------|
| LoginGate.exe | login_gate | Rust + tokio + egui |
| SelGate.exe | sel_gate | Rust + tokio + egui |
| RunGate.exe | run_gate | Rust + tokio + egui |
| LoginSrv.exe | login_srv | Rust + tokio + egui |
| DBServer.exe | db_server | Rust + tokio + egui + sqlx |
| M2Server.exe | m2_server | Rust + tokio + egui |
| LogServer.exe | log_server | Rust + tokio + egui |
| GameCenter.exe | game_center | Rust + tokio + egui |

**关键设计决策**：每个服务均为独立进程，拥有独立的egui图形界面，可独立启动、独立配置、独立监控。服务间通信采用自研内部二进制协议，非gRPC非HTTP，保持传统引擎的轻量级风格。

## 第二章：网关层详细设计（3个网关）

### 2.1 LoginGate（登录网关）

**核心职责**：接收客户端的TCP连接，处理账号登录请求的协议转发，将客户端登录请求转发至LoginSrv，将LoginSrv的验证结果返回给客户端。

**连接处理流程**：

1. 客户端发起TCP连接至LoginGate的监听端口
2. LoginGate分配连接会话，初始化会话状态
3. 客户端发送登录协议包（账号、密码、客户端版本）
4. LoginGate将登录请求转发至LoginSrv（内部通信）
5. LoginSrv进行账号验证，返回验证结果和登录令牌
6. LoginGate将验证结果转发回客户端
7. 客户端验证成功后，断开与LoginGate的连接，准备连接SelGate

**性能设计**：采用tokio异步IO模型，单网关支持10000并发连接。连接池化管理，空闲连接超时自动回收。

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、运行时长、监听端口、当前连接数 |
| 连接列表 | 连接IP、连接时间、当前状态（等待/已验证/异常） |
| 登录统计 | 今日登录次数、登录成功数、登录失败数、拒绝次数 |
| 实时日志 | 滚动显示登录请求日志（时间/来源IP/账号/结果） |
| 配置面板 | 监听地址、最大连接数、超时时间、登录服务地址 |
| 控制按钮 | 启动/停止服务、刷新连接、断开指定连接 |

### 2.2 SelGate（选服网关）

**核心职责**：接收已通过登录验证的玩家连接，提供区服列表查询、角色列表查询、角色创建/删除、角色选择功能。玩家选择角色后，返回RunGate的连接信息供客户端后续连接。

**连接处理流程**：

1. 客户端携带登录令牌连接SelGate
2. SelGate向LoginSrv验证令牌有效性
3. 验证通过后，向DBServer查询该账号的角色列表
4. 返回区服列表和角色列表给客户端
5. 客户端选择已有角色或创建新角色
6. SelGate向DBServer执行角色查询/创建操作
7. 返回所选角色的RunGate连接信息（IP、端口、进入令牌）
8. 客户端断开SelGate连接，准备连接RunGate

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、监听端口、当前连接数、令牌验证通过率 |
| 角色操作日志 | 时间/账号/操作类型（查询/创建/删除/选择）/角色名/结果 |
| 区服状态 | 各区服名称、在线人数、服务器状态、是否可注册新角色 |
| 配置面板 | 监听地址、最大连接数、令牌有效期、是否允许创建新角色 |
| 控制按钮 | 启动/停止服务、刷新区服状态 |

### 2.3 RunGate（运行网关）

**核心职责**：游戏运行期间的消息枢纽。接收客户端所有游戏内消息并转发至M2Server，接收M2Server的推送消息并转发给对应客户端。RunGate是所有游戏数据的必经之路，性能至关重要。

**消息转发机制**：

1. 客户端携带进入令牌连接RunGate
2. RunGate向M2Server验证令牌有效性，建立玩家会话
3. 客户端至M2Server方向：移动消息、技能消息、聊天消息、交易消息 → RunGate接收 → 转发至M2Server
4. M2Server至客户端方向：位置同步、战斗结果、聊天广播、物品变化 → M2Server推送至RunGate → RunGate转发至目标客户端
5. 客户端断开连接时，RunGate通知M2Server清理玩家会话

**关键性能指标**：

| 指标 | 设计目标 |
|-----|---------|
| 单网关最大连接数 | 5000玩家 |
| 消息转发延迟 | < 10毫秒 |
| 消息吞吐量 | 50000条/秒 |
| CPU占用率（满载）| < 40%单核 |
| 内存占用（满载） | < 200MB |

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、运行时长、监听端口、当前玩家数、今日进入人数 |
| 实时消息统计 | 每秒消息入/出数量、消息类型分布（移动/战斗/聊天/其他） |
| 性能指标 | CPU占用率、内存占用、网络带宽、平均延迟、最大延迟 |
| 在线玩家列表 | 玩家名、等级、所在地图、连接时长、最近消息时间 |
| 实时日志 | 滚动显示重要事件（玩家进入/退出、异常连接、网络波动） |
| 配置面板 | 监听地址、最大玩家数、消息队列大小、超时时间、M2Server地址 |
| 控制按钮 | 启动/停止服务、踢出玩家、全服公告、断开空闲连接 |

## 第三章：业务服务层详细设计（5个服务）

### 3.1 LoginSrv（登录服务）

**核心职责**：账号认证的核心逻辑服务。处理账号密码验证、登录令牌生成与验证、账号封禁检查、多端登录控制。

**业务流程**：

1. 接收LoginGate转发的账号密码验证请求
2. 向DBServer查询账号信息
3. 验证密码正确性（比对哈希值）
4. 检查账号状态（是否封禁、是否在线）
5. 生成会话令牌并记录令牌有效期
6. 返回验证结果给LoginGate

7. 接收SelGate转发的令牌验证请求
8. 验证令牌是否有效且未过期
9. 返回验证结果给SelGate

**数据安全设计**：

- 密码存储采用加盐哈希（Argon2id），不明文存储
- 登录令牌采用JWT风格的有状态令牌，令牌ID在服务端维护，可随时吊销
- 支持账号异地登录提醒和强制下线

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、监听端口、今日验证次数、验证通过率 |
| 账号统计 | 在线账号数、封禁账号数、近期注册趋势 |
| 封禁管理 | 封禁列表、解除封禁、临时封禁（可设时长）、封禁原因记录 |
| 实时日志 | 登录尝试日志（成功/失败/拒绝/异地登录） |
| 配置面板 | 令牌有效期、密码尝试次数限制、是否允许新账号注册 |
| 控制按钮 | 启动/停止服务、强制账号下线、导出登录统计 |

### 3.2 DBServer（数据库服务）

**核心职责**：所有服务的数据读写统一入口。提供数据缓存层、事务管理、ORM能力。M2Server、LoginSrv、SelGate、RunGate等所有服务均通过DBServer进行数据操作，不直接连接数据库。

**分层架构**：

```
┌─────────────────────────────────────────────────────────┐
│                     DBServer（进程内）                    │
├──────────────┬──────────────┬────────────────────────────┤
│  连接管理    │   数据缓存    │      ORM映射层              │
│  会话池      │  LRU缓存     │  实体对象<->数据库表映射      │
│  连接超时    │  缓存一致性  │  自动SQL生成                 │
└──────────────┴──────────────┴────────────────────────────┘
                           │ 数据库访问
                           ▼
                    ┌─────────────┐
                    │   MySQL     │
                    │   / Redis   │
                    └─────────────┘
```

**数据读写策略**：

- **玩家基础数据**（账号、角色、背包、装备）：读写均走DBServer，写操作同步持久化，读操作优先缓存
- **配置数据**（怪物配置、物品配置、地图配置、技能配置）：启动时加载到内存，只读访问
- **运行时数据**（玩家在线状态、临时buff、交易锁定）：纯内存，不持久化
- **日志数据**（玩家行为日志、GM操作日志）：异步写入，延迟容忍度高

**缓存失效策略**：
- 玩家退出游戏：清理该玩家所有缓存
- GM修改配置：清除对应配置缓存
- 定时刷新：低频配置每N分钟检查一次版本号

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、运行时长、数据库连接状态、Redis连接状态 |
| 连接池状态 | 总连接数、活跃连接数、空闲连接数、排队请求数 |
| 缓存统计 | 缓存命中率、缓存大小、最近缓存失效时间分布、Top10热点Key |
| SQL统计 | 每秒查询数、慢查询列表（耗时>100ms）、SQL类型分布 |
| 数据查询 | 按账号查角色、按角色查物品、SQL调试窗口（需权限） |
| 实时日志 | 数据库操作日志、慢查询告警、连接异常 |
| 配置面板 | MySQL地址端口账号、Redis地址、缓存大小、连接池大小、超时时间 |
| 控制按钮 | 启动/停止服务、刷新所有缓存、清理指定缓存、数据库备份 |

### 3.3 M2Server（主引擎服务）

**核心职责**：游戏逻辑的核心引擎。处理所有地图管理、实体管理、战斗系统、技能系统、物品系统、NPC系统、怪物AI、行会系统、任务系统、脚本系统。M2Server是整个引擎的心脏，其他所有服务围绕它工作。

**内部模块划分**：

```
M2Server进程
├─ 地图管理器（MapManager）
│   ├─ 地图加载（从配置文件读取地图数据）
│   ├─ 地图实例管理（每个地图一个独立实例）
│   └─ 地图间跳转/传送处理
│
├─ 实体系统（EntitySystem）
│   ├─ 玩家实体（Player）
│   ├─ 怪物实体（Monster）
│   ├─ NPC实体
│   ├─ 宠物/召唤物实体
│   └─ 物品掉落实体
│
├─ 战斗系统（CombatSystem）
│   ├─ 伤害计算公式
│   ├─ 技能释放与效果
│   ├─ 仇恨与AI
│   └─ 战斗状态管理
│
├─ 物品系统（ItemSystem）
│   ├─ 背包管理
│   ├─ 装备穿戴
│   ├─ 物品使用
│   ├─ 物品交易
│   └─ 商店系统
│
├─ 社交系统（SocialSystem）
│   ├─ 聊天频道管理
│   ├─ 行会管理
│   ├─ 好友系统
│   └─ 组队系统
│
├─ 脚本引擎（ScriptEngine）
│   ├─ NPC对话脚本
│   ├─ 任务脚本
│   ├─ 活动脚本
│   └─ GM指令脚本
│
└─ 定时任务（Scheduler）
    ├─ 怪物刷新
    ├─ 活动定时开启/关闭
    ├─ 每日重置
    └─ 数据自动保存
```

**线程模型设计**：

| 线程类型 | 职责 | 数量 |
|---------|------|------|
| 地图主线程 | 处理该地图内所有实体的更新（移动、战斗、AI） | 每个地图1个 |
| 全局调度线程 | 处理跨地图事务、定时任务、全局状态更新 | 1个 |
| 消息接收线程 | 从RunGate接收玩家消息，分发至对应地图线程 | 1个 |
| 数据保存线程 | 定时将玩家数据写入DBServer | 1个 |

**关键性能考量**：
- 地图间无共享状态，天然支持并行处理
- 玩家消息异步分发，避免阻塞
- 战斗计算无锁化，采用消息串行化保证一致性
- AI状态机设计，每帧仅处理需要更新的实体

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、运行时长、当前在线人数、当前地图数、怪物总数 |
| 地图监控 | 地图列表、每个地图的玩家数/怪物数/NPC数、地图状态 |
| 在线玩家列表 | 玩家名、等级、职业、所在地图、坐标、当前HP/MP、金币 |
| 战斗统计 | 每秒战斗数、技能释放频率、伤害分布、死亡统计 |
| 性能指标 | CPU占用、内存占用、单帧处理时长、消息队列长度 |
| GM指令控制台 | GM命令输入窗口、命令执行结果、命令历史记录 |
| 物品/怪物查询 | 物品配置查询、怪物配置查询、GM刷怪/刷物 |
| 实时日志 | 服务运行日志、玩家关键行为日志、错误日志 |
| 配置面板 | 服务器端口、最大在线人数、地图列表配置、经验倍率、掉落倍率 |
| 控制按钮 | 启动/停止服务、全服公告、手动保存数据、重启指定地图 |

### 3.4 LogServer（日志服务）

**核心职责**：集中化日志收集与存储。所有其他服务（网关和业务服务）均通过内部协议向LogServer发送日志消息，LogServer统一写入日志文件并提供查询能力。

**日志类型**：

| 日志类型 | 来源服务 | 存储方式 |
|---------|---------|---------|
| 系统运行日志 | 所有服务 | 按日滚动文件 |
| 玩家行为日志 | M2Server | 按日分区数据库表 |
| 登录登出日志 | LoginGate/SelGate/RunGate | 数据库表 |
| 交易日志 | M2Server | 数据库表 |
| 物品获得/消耗日志 | M2Server | 数据库表 |
| GM操作日志 | M2Server/GameCenter | 数据库表，不可删除 |
| 异常/错误日志 | 所有服务 | 独立错误日志文件 |

**egui管理界面内容**：

| 界面区块 | 显示内容 |
|---------|---------|
| 服务器状态 | 运行状态、监听端口、今日日志条数、日志写入速度 |

## 第四章：服务间通信协议设计

### 4.1 协议设计原则

**原则一：简洁二进制**。采用自研二进制协议，轻量级、低开销、易解析。不使用gRPC/HTTP等重型协议，保持传统引擎的轻量风格。

**原则二：请求响应模型**。除日志类消息外，所有业务通信采用请求-响应模型，确保调用者可等待结果。

**原则三：消息自描述**。每个消息包含消息类型ID和消息长度，接收方可根据类型ID选择正确解析方式。

**原则四：超时与重试**。所有服务间通信设置超时时间，超时后按业务特性决定是否重试或报错。

### 4.2 通信矩阵

|---------------|----------|---------|---------|---------|---------|---------|----------|------------|

| **SelGate** | 无 | — | 无 | 令牌验证 | 角色查询/创建 | 无 | 日志 | 心跳/状态 |
| **RunGate** | 无 | 无 | — | 无 | 玩家数据读写 | 消息转发 | 日志 | 心跳/状态 |
| **LoginSrv** | 无 | 无 | 无 | — | 账号查询/更新 | 无 | 日志 | 心跳/状态 |
| **DBServer** | 无 | 无 | 无 | 无 | — | 无 | 日志 | 心跳/状态 |
| **M2Server** | 无 | 无 | 消息推送 | 在线状态通知 | 玩家数据保存/加载 | — | 日志 | 心跳/状态 |

## 第五章：客户端与网关通信协议

### 5.1 协议框架

客户端与三个网关之间的通信采用统一的协议框架，但不同网关支持的消息类型不同。

**消息格式**：

| 字段 | 长度 | 说明 |
|-----|------|------|
| 包头标识 | 2字节 | 固定值0xAABB，用于粘包识别 |
| 消息长度 | 4字节 | 消息体长度（不含包头和长度字段本身） |
| 消息类型 | 2字节 | 消息类型ID，决定如何解析消息体 |
| 序列号 | 4字节 | 请求序列号，用于匹配请求和响应 |
| 时间戳 | 4字节 | Unix时间戳，防重放 |
| 消息体 | N字节 | 根据消息类型解析的结构化数据 |
| 校验码 | 2字节 | CRC16校验，覆盖长度字段+类型字段+消息体 |

**加密策略**：

- 登录握手阶段采用非对称加密（RSA）交换对称密钥
- 游戏通信阶段采用对称加密（AES-128-CFB），每N小时轮换密钥
- 密钥在服务端生成，客户端被动接收

### 5.2 LoginGate协议（登录阶段）

| 消息类型 | 方向 | 内容说明 |
|---------|------|---------|
| LOGIN_REQUEST | 客户端→LoginGate | 账号、密码哈希、客户端版本号 |
| LOGIN_RESPONSE | LoginGate→客户端 | 登录结果（成功/失败/封禁）、令牌、区服列表 |
| SERVER_LIST_REQUEST | 客户端→LoginGate | 请求区服列表更新 |
| SERVER_LIST_RESPONSE | LoginGate→客户端 | 区服名称、区服状态、区服在线人数 |

### 5.3 SelGate协议（选服阶段）

| 消息类型 | 方向 | 内容说明 |
|---------|------|---------|
| ENTER_REQUEST | 客户端→SelGate | 登录令牌、区服选择 |
| ENTER_RESPONSE | SelGate→客户端 | 令牌验证结果、角色列表 |

### 7.2 数值体系设计

**伤害计算公式**（物理攻击）：

```
最终伤害 = (攻击者攻击力 - 被攻击者防御力/2) × 技能系数 × 随机波动(0.9~1.1)
```

说明：防御力采用减伤机制，防御力越高减伤越明显，但存在最低伤害保底。

**魔法攻击伤害公式**：

```
最终魔法伤害 = (攻击者魔法攻击 - 被攻击者魔法防御/3) × 技能系数 × 魔防穿透修正
```

**经验获得公式**：

```
获得经验 = 怪物基础经验 × 怪物等级修正 × 队伍人数修正 × 活动倍率
```

**等级-经验曲线**（经典）：

```
升级所需经验 = 基础值 × 当前等级^2.5
```

**掉落概率公式**：

```
单个物品掉落概率 = 基础掉落率 × (1 + 幸运值/200) × 活动倍率
```

说明：每个怪物的每个可掉落物品独立判定概率，可同时掉落多个物品。

### 7.3 经典玩法特性还原要点

**秒卡/蓝药系统**：

- HP药水和MP药水有冷却时间（如1秒）
- 使用后立即恢复指定数值
- 药水占用背包格子
- 药水有等级限制（高等级药水需一定等级才能使用）

**装备强化系统**：

- 强化消耗金币和强化石
- 强化等级上限（如+15）
- 强化成功率随等级下降
- 强化失败惩罚（等级降低/保持不变/装备消失，可配置）
- 强化属性加成线性或递增

**PK惩罚系统**：

- 击杀白名玩家增加PK值
- PK值超过阈值变为黄名/红名
- 红名玩家死亡有较高概率掉落装备
- 红名玩家不可进入部分安全区
- PK值随在线时间缓慢减少

**沙巴克攻城战**：

- 每周固定时间开启
- 行会报名参与
- 攻城战期间沙巴克地图进入战争状态
- 占领机制：占领旗帜并持续持有至战争结束
- 胜利行会获得专属称号、每日奖励、税收特权

**自动寻路**：

- 客户端基于地图阻挡数据计算路径
- 服务端定期校验玩家位置合理性（防瞬移外挂）

## 第八章：M2Server内部引擎架构详解

### 8.1 地图引擎设计

**地图数据结构**：

每个地图独立管理以下数据：
- 地图元信息（地图ID、名称、类型、是否PK区）
- 2D网格阻挡数据（每格是否可通行、是否可放置物品）
- 玩家实体列表（当前在该地图的所有玩家）
- 怪物实体列表
- NPC实体列表
- 掉落物品列表
- 事件触发器列表（进入/离开某区域触发）

**地图更新循环**（固定频率，如每秒10帧）：

1. 处理所有地图内消息队列（来自玩家的移动/技能/交互请求）
2. 更新怪物AI状态
3. 处理技能效果和伤害结算
4. 更新buff/debuff状态和倒计时
5. 处理定时器到期事件（如定时刷新怪物、活动开始/结束）
6. 生成需要同步给玩家的位置/状态更新消息
7. 定期检查玩家位置合理性（防外挂）

### 8.2 实体系统

**实体基类属性**（所有实体共享）：

- 实体ID：全局唯一标识
- 实体类型：玩家/怪物/NPC/掉落物
- 坐标：X, Y
- 所在地图ID
- 方向：8方向
- HP/MP：生命值/魔法值

**玩家实体特有属性**：

- 角色基础信息（职业、等级、经验）
- 属性面板（力量/敏捷/体力/智力及衍生属性）
- 装备栏状态
- 背包和仓库
- 当前目标（战斗目标、交易目标、对话NPC）
- 技能状态（冷却中的技能）
- buff/debuff列表
- 好友/行会/队伍信息引用
- 客户端连接信息（网关引用）

**怪物实体特有属性**：

- 怪物配置ID引用
- AI状态机当前状态
- 当前仇恨目标
- 巡逻路径
- 刷新点和刷新倒计时
- 掉落列表引用

### 8.3 战斗引擎

**战斗触发流程**：

1. 玩家点击攻击目标或释放技能
2. RunGate将消息转发至M2Server
3. M2Server校验：目标是否存在、距离是否合法、技能是否可用（冷却/MP）
4. 计算伤害（根据公式）
5. 目标HP扣减
6. 判断是否死亡
7. 若死亡：处理掉落、经验分配、死亡状态
8. 将战斗结果广播给视野内玩家

**伤害计算引擎**：

- 支持物理伤害和魔法伤害分离计算
- 支持暴击、闪避、命中等概率事件
- 支持buff/debuff的实时效果叠加
- 支持技能特殊效果（减速、眩晕、中毒、吸血等）

### 8.4 脚本引擎

**脚本用途**：

- NPC对话逻辑（多分支选择、条件判断）
- 任务系统（任务接受条件、任务进度跟踪、任务奖励发放）
- 活动脚本（定时开启的活动、特殊事件逻辑）
- 物品使用效果（特殊物品的使用效果）

**脚本引擎选择**：

采用Lua作为脚本语言，理由：
- Lua在游戏行业广泛使用，资料丰富
- 性能优秀，JIT模式下接近原生代码
- 与Rust集成成熟（mlua/rlua库）
- 轻量级，脚本编写门槛低

**Rust↔Lua互操作**：

- Rust向Lua暴露API（如"获取玩家金币"、"传送玩家"、"发放物品"、"发送消息"）
- Lua脚本中调用这些API实现业务逻辑
- 脚本热更新：修改脚本后无需重启服务即可生效

## 第九章：部署与运维

### 9.1 部署形态

**单机部署**（推荐中小运营）：

8个服务全部运行在一台物理机或云服务器上。配置要求：
- CPU：8核以上
- 内存：16GB以上
- 硬盘：100GB SSD以上（数据库和日志）
- 网络：100Mbps上行带宽（5000人同时在线规模）

**多机部署**（推荐大型运营）：

- 机器1：LoginGate、SelGate、RunGate（网关集群，带宽要求高）
- 机器2：M2Server（主引擎，CPU和内存要求高）
- 机器3：DBServer + MySQL（数据库，IO性能要求高）
- 机器4：LoginSrv + LogServer + GameCenter（轻量服务）

**容器化部署**（可选）：

每个服务打包为独立Docker镜像，通过docker-compose或Kubernetes管理。不推荐初学者使用，增加运维复杂度。

### 9.2 启动顺序

```
1. LogServer    ← 首先启动日志服务（所有服务依赖日志）
2. DBServer     ← 其次启动数据库服务（所有业务依赖数据）
3. GameCenter   ← 启动管理中心
4. LoginSrv     ← 启动登录服务
5. M2Server     ← 启动主引擎（等待地图和配置加载完成）
6. RunGate      ← 启动运行网关（需等M2Server就绪）
7. SelGate      ← 启动选服网关
8. LoginGate    ← 最后启动登录网关（对外接入点）
```

**停止顺序为启动顺序的逆序**。GameCenter可一键按顺序启停所有服务。

### 9.3 配置文件

每个服务拥有独立的配置文件（TOML格式），存放于config/目录：

| 配置文件 | 用途 |
|---------|------|
| game_center.toml | 管理中心配置（其他服务地址、管理端口、管理员账号） |
| log_server.toml | 日志服务配置（监听端口、日志目录、保留天数、数据库连接） |
| db_server.toml | 数据库服务配置（MySQL/Redis连接、缓存大小、连接池参数） |
| login_srv.toml | 登录服务配置（监听端口、令牌有效期、密码策略） |
| m2_server.toml | 主引擎配置（监听端口、最大在线、地图列表、经验/掉落倍率、GM指令权限） |
| login_gate.toml | 登录网关配置（监听端口、最大连接数、LoginSrv地址） |
| sel_gate.toml | 选服网关配置（监听端口、最大连接数、LoginSrv/DBServer地址） |
| run_gate.toml | 运行网关配置（监听端口、最大玩家数、M2Server地址、超时时间） |

所有配置可在GameCenter中修改并热生效。

### 9.4 监控与告警

**监控指标体系**：

| 监控层级 | 指标 | 告警阈值 |
|---------|------|---------|
| 系统级 | CPU占用、内存占用、磁盘空间、网络带宽 | CPU>80%、内存>85%、磁盘<10%可用 |
| 服务级 | 服务存活状态、消息队列长度、响应延迟 | 服务无响应、队列>1000条、延迟>100ms |
| 业务级 | 在线人数、登录成功率、错误率异常 | 登录成功率<90%、错误率>5% |
| 数据级 | 数据库连接池使用率、Redis命中率 | 连接池>90%、命中率<80% |

**告警通知**：

- GameCenter界面告警（所有运营人员可见）
- 邮件通知（可选）
- 短信/即时消息通知（可选，需对接第三方服务）

### 9.5 备份与恢复

**自动备份策略**：

| 数据类型 | 备份频率 | 保留周期 | 存储位置 |
|---------|---------|---------|---------|
| 玩家数据（角色/背包） | 每日凌晨2点全量备份 + 每小时增量快照 | 30天 | 本地磁盘 + 异地存储 |
| 配置数据（物品/怪物/地图） | 每次GM修改后自动版本化 | 永久 | 版本控制系统 |
| 日志数据 | 按日滚动，归档 | 90天 | 本地磁盘 |
| 数据库全库 | 每日凌晨2点 | 7天 | 本地磁盘 + 异地存储 |

**手动备份触发**：运营人员可在GameCenter界面手动触发任意类型的备份。

**恢复流程**：

1. 在GameCenter选择恢复点（按时间列出所有备份）
2. 选择恢复范围（全库/指定玩家/指定表）
3. 确认后进入恢复流程（自动停止相关服务→恢复数据→重启服务）
4. 恢复完成后自动校验数据完整性

## 第十章：技术选型总结与实施建议

### 10.1 最终技术选型

| 层面 | 选型 | 理由 |
|-----|------|------|
| 编程语言 | Rust | 内存安全、无GC、高性能、适合长运行服务器程序 |
| 异步运行时 | tokio | Rust生态最成熟的异步运行时，性能经过大规模验证 |
| 服务界面 | egui | 纯Rust实现的即时模式GUI，与主程序同进程运行，美观高效 |
| 数据库（主） | MySQL 8.0+ | 成熟稳定、团队熟悉度高、事务支持完善 |
| 缓存/内存存储 | Redis 7.0+ | 高性能KV存储，支持丰富数据结构 |
| 脚本引擎 | Lua 5.4 | 游戏行业标准脚本语言，与Rust集成成熟 |
| 服务间通信 | 自研二进制协议 | 轻量、低延迟、完全可控，保持传统引擎风格 |
| 客户端引擎 | Unity 2022 LTS+ | 成熟2D/轻度3D能力、跨平台发布、团队容易招聘 |
| 客户端GUI | Unity UGUI/UI Toolkit | 引擎内置方案，上手快 |

### 10.2 与传统引擎的差异对比

| 对比项 | 传统引擎（C++/Delphi） | 本方案（Rust） |
|-------|----------------------|--------------|
| 内存管理 | 手动管理（易泄漏）/ 部分GC | 所有权系统编译期保证 |
| 并发模型 | 多线程+锁（易死锁） | tokio异步+清晰消息边界 |
| 网络IO | 完成端口/IOCP（Windows特定） | tokio异步（跨平台） |
| 界面 | WinAPI/MFC/VCL（老旧、难维护） | egui（现代、易扩展） |
| 脚本引擎 | 自研PaxScript或Lua | Lua（行业标准） |
| 数据库 | 自建连接池（SQL Server/MySQL） | sqlx异步ORM |
| 跨平台 | Windows only | 原生支持Windows/Linux/macOS |
| 性能 | 高（C++手动优化） | 高（Rust零成本抽象） |
| 安全 | 空指针/野指针/缓冲区溢出风险高 | 编译期消除内存安全问题 |
| 热更新 | 需重启服务（部分引擎支持脚本热更新） | 支持脚本热更新 + 配置热更新 |

### 10.3 实施路线图

**阶段一：基础框架搭建（2周）**

- 8个服务的基础程序框架（空壳，可启动可停止）
