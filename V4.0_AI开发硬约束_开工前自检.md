# V4.0 AI 开发硬约束（开工前自检清单）

> **适用范围**：服务端开发、客户端开发、协议设计、数据库设计、功能文档撰写。  
> **生效时间**：每次 AI 开始写代码/写设计/改协议前，必须先通读本约束。  
> **核心原则**：经典玩法不变（对外呈现），内部实现全现代化（工程化/性能/安全）。

---

## 一、引擎身份与版本识别（开工第一步）

### 1.1 当前版本识别

- 本项目引擎版本 = **V4.0**（基于传统 8 服务架构 + Rust 全栈 + ECS + Bevy 客户端）
- 文档基准 = [OpenMir2_Rust现代化引擎技术文档_V4.0.md](file:///workspace/demo-project/OpenMir2_Rust%E7%8E%B0%E4%BB%A3%E5%8C%96%E5%BC%95%E6%93%8E%E6%8A%80%E6%9C%AF%E6%96%87%E6%A1%A3_V4.0.md)
- 禁止自作主张：切换到 V1.0/V2.0/V3.0 的任何一个旧设计
- 禁止自作主张：把客户端换成 Unity / Unreal
- 禁止自作主张：把服务端主架构换成面向对象的传统模型

### 1.2 三个"我是谁"必答（写代码前自检）

- Q1：**当前项目的服务端主架构是什么？**
  - A：**ECS（Entity-Component-System）**，数据与行为分离
  - ❌ 错误答案：类继承（Player 类、Monster 类、Entity 基类）
- Q2：**当前项目的客户端用什么引擎？**
  - A：**Bevy（纯 Rust + wgpu）**，与服务端共享 crate
  - ❌ 错误答案：Unity / Unreal / 自研 C++
- Q3：**当前项目保留的经典玩法是什么？**
  - A：**三职业（战/法/道）+ 经典传奇属性（DC/MC/SC/AC/MAC）+ 经典伤害公式 + 经典掉落 + 经典沙巴克**
  - ❌ 错误答案：通用 MMO 属性系统 / 多职业自由组合

如果上面三个问题答不出任何一个 → **停止写代码，先重看 V4.0 文档第一至第六章**。

## 二、服务端架构硬性红线（不可违背）

### 2.1 8 服务必须严格保留

**8 个服务的名称与职责固定不变**：

| 服务 | 职责关键字 | 新增/重命名 = 违反红线 |
|-----|----------|-------------------|
| LoginGate | 登录接入（TCP） | 禁止合并进 SelGate / M2Server |
| SelGate | 选服/选角色接入 | 禁止合并进 LoginGate / RunGate |
| RunGate | 游戏运行消息枢纽 | 禁止直接执行业务逻辑（必须转发 M2Server） |
| LoginSrv | 账号认证/令牌管理 | 禁止与 DBServer 合并 |
| **M2Server** | **核心游戏逻辑（地图/战斗/技能/掉落/活动）** | **禁止拆分（未来分布式拆分为后续版本规划，非 V4.0）** |
| DBServer | MySQL + Redis 统一入口 + 缓存 | 禁止服务直连数据库 |
| LogServer | 集中日志采集与查询 | 禁止业务日志散落 |
| GameCenter | 统一运维/GM/配置管理/数据回滚 | 禁止分散到各服务自行实现 |

**开工自检**：如果 AI 准备写"新服务"或"合并两个服务" → 先回答"为什么 V4.0 的 8 服务不够用？" → 无法给出理由 → **停止设计**。

### 2.2 ECS 架构硬规则

**任何新增游戏逻辑都必须落入 ECS 模式**：

- 新增一个**数据集合** → 定义一个 **Component**（纯数据 struct）
- 新增一个**行为逻辑** → 定义一个 **System**（纯函数，输入只读 Components，输出修改/事件）
- 新增一个**实体种类** → **Entity + Tag + 组合 Component**（如玩家=Entity+PlayerTag+Position+Equipment；怪物=Entity+MonsterTag+DropTable）
- **禁止在 Component 里写方法/行为**
- **禁止在 System 内部持有可变状态**
- **禁止手写类继承（Player 继承 Entity 这种写法禁止）**

**正确范例**：
- 要做"装备强化" → 新增 `EnhanceEvent`（Service 层消息） → `InventorySystem` 读取装备、金币、强化石 Components → 判定 → 修改装备 `EnhanceLevelComponent` → 写回 DB
- 要做"沙巴克旗帜" → 新增 `FlagEntity(Entity) + FlagStateComponent + CaptureTimerComponent` → `SaccoonSystem` 每帧计算旗帜被哪方占领

**错误范例（如果 AI 写出下列任何一种，必须推翻重写）**：
- 写 `class Player : public Entity { void Move(); void Attack(); }` 这种面向对象继承
- 把"战斗"写在玩家对象内部的方法里
- 新增"全局类/单例"承载业务逻辑

### 2.3 服务间通信协议

- **Protobuf（prost）** 统一，禁止手写二进制协议
- **字段编号**一旦定义，不可重排、不可删除后复用编号（可标记 deprecated）
- 每次协议改动 → 必须同时更新 `mir2-shared/proto/*.proto` 与客户端/服务端双端生成代码

## 三、经典玩法硬性保留（不可改造/现代化）

### 3.1 三职业（战士/法师/道士）

- 职业数量固定 = **3**，任何"新增第 4 职业"的提议都是 V4.0 之外
- 每个职业有**独立技能树**（烈火剑法 / 冰咆哮 / 召唤骷髅等），不可"自由学习全职业技能"
- 转职/多职业切换 = 禁止（V4.0 不支持）

### 3.2 经典传奇属性体系

**属性命名固定为以下集合，任何新增属性必须给出传奇经典出处**：

- DC / MC / SC / AC / MAC / HP / MP / 准确 / 敏捷 / 幸运 / 诅咒 / 负重

**伤害公式固定（不可替换为其他游戏公式）**：
- 物理伤害 = `random(DC下限, DC上限) × 技能倍率 - 目标 AC/2`
- 魔法伤害 = `random(MC下限, MC上限) × 技能系数 - 目标 MAC/3`
- 道术伤害 = `random(SC下限, SC上限) × 技能系数 - 目标 MAC/3`
- 幸运 / 暴击 / 命中 / 闪避 = 按 V4.0 文档定义的经典数值规则

**AI 自检**：如果准备写"平衡/重写数值" → 必须从 V4.0 第 3 章和第 4 章给出的数值基准出发，禁止凭空引入 WoW/RO/原神风格公式。

### 3.3 沙巴克、强化、掉落等"经典玩法"

- 沙巴克：**宣战期(20:00-20:30) → 战争期(20:30-21:30) → 休战期** 固定节奏，且必须有"旗帜占领机制"（3 面旗帜+倒计时）
- 强化：**+1~+15**，必须遵循经典"失败降级/装备消失"规则；必须有金币/强化石消耗；继承/分解功能保留
- 掉落：怪物独立掉落配置表 + 独立随机 + 拾取归属保护期

## 四、Rust 语言与工程化约束

### 4.1 技术栈固定

| 模块 | 技术选型 | 禁止替换为 |
|------|---------|-----------|
| 语言 | Rust（edition 2021+） | C / C++ / Go / Java / Python / Zig |
| 异步运行时 | tokio | async-std / smol / 手写 epoll |
| 服务端界面 | egui | 控制台 printf / WinForm / Tauri |
| 数据库 ORM | sqlx（异步 SQL） | Diesel（除非已明确在文档里说明） |
| 脚本引擎 | Lua（mlua 5.4） | Python / WASM / JavaScript |
| 客户端渲染 | Bevy + wgpu | Unity / Unreal / Godot |

### 4.2 共享 crate 优先

任何数据结构/协议/常量/公式：**先考虑是否放入 `mir2-shared`**，让服务端与客户端同一份定义。

- Protobuf 定义 → `mir2-shared/proto/`
- Components 定义 → `mir2-shared/components/`
- 配置常量 → `mir2-shared/constants/`
- 公式实现 → `mir2-shared/calc/`

**禁止两边各写一份同构代码**。

### 4.3 unsafe 禁止

除非 V4.0 文档明确列出例外（与 FFI / Lua 交互等），否则禁止在 Rust 代码里写 `unsafe`。

如果发现"必须写 unsafe 才能实现" → 停下来，回到架构层面重新设计，而不是写 `unsafe`。

## 五、反外挂安全硬约束

### 5.1 客户端上报数据 = 全不可信

**下列任何一项，只要 AI 写了"服务端信任客户端上报值"，都应立即推翻重写**：

- ❌ 信任客户端发送的"伤害数字"
- ❌ 信任客户端发送的"位置坐标"
- ❌ 信任客户端发送的"技能释放成功"（应在服务端独立校验冷却/MP/范围）
- ❌ 信任客户端发送的"交易物品列表"
- ❌ 信任客户端发送的"经验/金币/道具增量"

**正确做法**：

1. 客户端仅上报"意图"（我想攻击谁 / 我想移动到哪 / 我想交易什么）
2. 服务端独立校验：范围、冷却、物品归属、金币数量、目标存活状态
3. 服务端独立计算：伤害、掉落、经验、货币变动
4. 服务端写入数据库并把结果同步回客户端

### 5.2 异常行为检测必做

- 移动瞬移（单位时间位移异常）
- 攻击频率异常（攻速被加速挂修改）
- 秒怪（单玩家单时段击杀数量异常）
- 高频交易/高频金币进出
- 操作间隔完全规律（机器人特征）

### 5.3 支付与 GM 操作必须有审计日志

任何涉及"充值、GM 修改玩家、回滚数据"的操作 → **必须在 LogServer 写一条不可修改的审计日志**。

## 六、热更新与版本兼容硬约束

### 6.1 Lua 脚本热更新

- NPC 对话脚本、活动脚本、任务脚本、物品使用效果脚本 → 必须走 Lua 脚本层
- 禁止把"活动逻辑、NPC 对话"写死在 Rust 硬编码里
- 新增脚本类型 → 要同时提供热加载流程（文档说明 GM 如何一键更新）

### 6.2 配置表热更新

- 物品/技能/怪物/地图配置以文件（TOML / JSON / Lua table）加载
- 新增配置项必须可热重载（不重启 M2Server 生效）
- 正在战斗/交易中的玩家"沿用旧配置直到下一状态切换点"，禁止战斗中属性跳变

### 6.3 客户端版本兼容

- 每次协议变更，**必须保持 Protobuf 字段向后兼容**（新增字段，不重排，不删除已发布字段）
- 必须设计"客户端版本号 < 最低兼容版本 → 强制更新提示"
- 同一服务端版本必须支持 ≥2 个历史客户端版本（允许"降级表现"，但不可崩溃/直接掉线）

## 七、数据库与数据安全硬约束

### 7.1 数据库入口单一化

- 任何服务（LoginSrv / M2Server / GameCenter / LogServer）**禁止直连 MySQL / Redis**
- 统一走 **DBServer**：查询/写入/事务/缓存层全在 DBServer 内完成
- 禁止绕过 DBServer：如果某功能"需要写库" → 在 DBServer 里新增该业务的查询接口，再在调用服务调用它

### 7.2 备份与回滚

- 全量备份 = **每日 02:00**
- 增量快照 = **每小时**
- 回滚必须 GM 确认 → 写入审计日志 → 事务执行 → 通知玩家
- 禁止"手动改数据库表"作为日常运维方式（GM 工具必须可视化界面执行）

### 7.3 合服流程可重复

- 合服必须做成**自动化流程**（可反复测试/回滚/重执行）
- 合服冲突处理规则：角色名/行会名冲突 → 保留先创建者，后创建者强制改名界面
- 禁止一次性手工 SQL 脚本（不可复现 = 禁止上生产）

## 八、开工自检 12 问（写代码/写设计前，必须过一遍）

**在你准备写任何代码/方案前，按顺序自问自答**：

1. □ 我知道这是 V4.0，8 服务 + Rust + ECS + Bevy，对吗？
2. □ 我写的东西放在 8 个服务里的哪一个？职责是否符合该服务的定位？
3. □ 如果是游戏逻辑（战斗/移动/技能/NPC），我是不是走 ECS（新增 Component/System，而不是手写类继承）？
4. □ 如果我新增了数据结构，它是否进了 `mir2-shared` 共享 crate？
5. □ 如果我修改协议，我是否保留字段编号不变、用 Protobuf 定义，并且在两边生成代码？
6. □ 我写的功能是否会导致"客户端传值=服务端信任"？如果是，我是否改为服务端独立校验并计算？
7. □ 如果影响数值（伤害/经验/掉落/金币），我用的是传奇经典公式吗？
8. □ 如果影响职业平衡，我是否从战/法/道三职业定位出发，而不是凭空引入新机制？
9. □ 数据库访问是不是走 DBServer（而不是服务直连 MySQL）？
10. □ 是否有审计日志覆盖（GM、支付、数据回滚、异常行为封号）？
11. □ 我的改动是否需要停机？如果是，我有没有提供 Lua/配置热更新替代方案？
12. □ 如果玩家同时在线 5000+，我这段代码的性能预期（CPU/内存/IO）合理吗？

**如果上面任意一个问题答不出 = 停止写代码，先回到 V4.0 文档对应章节补认知，再开工。**
