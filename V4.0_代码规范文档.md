### 2.2 组件命名规范

- **组件名必须是数据，不是行为**：正确示例 `Position`、`Health`、`Cooldowns`；错误示例 `Player`、`Move`
- **Tag 组件**（仅用于筛选某类实体）：后缀为 `Tag`，例如 `PlayerTag`、`MonsterTag`、`NpcTag`、`ItemDropTag`
- **状态组件**：使用枚举变体表达状态，例如 `AIState::Idle / AIState::Chase / AIState::Attack`
- **所有组件必须派生 `Clone + Debug`**；如无充分理由，不要派生 `PartialEq / Eq`（频繁比较开销大）

```rust
// 正确示例
#[derive(Clone, Debug)]
pub struct Position {
    pub map_id: u32,
    pub x: u16,
    pub y: u16,
}

#[derive(Clone, Debug, Default)]
pub struct Health {
    pub current: u32,
    pub max: u32,
    pub regen_per_sec: u32,
}

// 正确：Tag 组件单元结构体
#[derive(Clone, Debug, Default)]
pub struct PlayerTag;
```

### 3.2 System 函数签名规范

```rust
// 规范：System 函数只依赖 World / Resources / 事件，不持有任何内部可变状态
//
// 正确示例：
pub fn tick_movement(
    world: &mut World,              // 可读写的 World
    resources: &Resources,          // 只读资源（时间、随机数、地图数据…）
    event_queue: &mut EventQueue,   // 事件队列（写入移动结果/聊天消息）
    now_ms: u64,                    // 当帧时间
)

// 禁止任何 System 内部写 static mut / 隐藏状态；所有状态必须放入 Components
```

### 3.3 System 内部行为规范

每个 System 必须做的三件事：

1. **读取需要的 Components**（通过查询/筛选）
2. **执行业务逻辑**（纯计算/查表/调用脚本/调用公式）
3. **把结果写回 Components 或写入事件队列**（不直接调用网络发送）

```
│ 错误做法：System 里直接调用 "socket.send(…)" 给客户端
│ 正确做法：System 把结果（"玩家X移动到(100,200)"）写进事件队列，
│         由独立的 network 模块在 tick 结束后统一打包发给 RunGate
```

## 四、Protobuf 协议规范

### 4.1 文件组织

```
mir2-shared/src/proto/
├── login.proto      ← 登录/认证
├── player.proto     ← 玩家数据/选服/选角色/角色创建
├── game.proto       ← 游戏消息（移动/战斗/技能/聊天/背包）
├── gm.proto         ← GM 指令
└── inter_svc.proto  ← 8 个服务之间的内部 RPC（M2Server→DBServer 等）
```

### 4.2 字段编号与兼容性规则

- **每个 .proto 第 1 行必须写 `syntax = "proto3";`**
- **字段编号一旦发布，不可重排、不可删除后复用**。废弃字段应加 `// deprecated` 注释，编号保留
- **新增字段必须使用全新的、未使用过的编号**
- 不要使用 `required`（proto3 已移除；proto2 下也避免）
- 字符串字段：中文必须 UTF-8（Protobuf 原生编码），协议内不用 GBK

```protobuf
// 正确示例
syntax = "proto3";

package mir2.login;

message LoginRequest {
    string account = 1;
    string password_sha256 = 2;  // 密码走哈希，不可明文
    uint32 client_version = 3;
}

message LoginResponse {
    uint32 error_code = 1;
    uint64 session_token = 2;
    string char_list = 3;         // 由 client 解析内部结构
    // field 4 保留将来扩展；不可把"4"挪作他用
}
```

### 4.3 消息类型命名规范

- 请求：`XxxRequest`
- 响应：`XxxResponse`
- 通知（服务端主动推送）：`XxxNotice`
- 错误：`XxxError`（或直接在 `error_code` 字段内表达）

## 五、数据库与 DBServer 代码规范

### 5.1 所有服务禁止直连数据库

```
│ 错误做法：LoginSrv 直接 sqlx::Pool::connect("mysql://...")
```

## 七、Rust 代码风格约束

本项目使用 Rust 社区默认规范 + 少量强约束：

| 规则 | 说明 |
|------|------|
| `cargo fmt` 格式化 | 提交前必须执行；CI 中 `cargo fmt --check` 校验 |
| `cargo clippy --all-targets --all-features` | 必须 0 warning；warning 视作错误，禁止提交 |
| 命名 | 类型/枚举 `UpperCamelCase`；函数/方法/字段 `snake_case`；常量 `SCREAMING_SNAKE_CASE` |
| 模块名 | 与文件/目录名一致，小写 + 下划线 |
| 可见性 | 默认 `pub(crate)`，确需跨 crate 才用 `pub`；最小暴露原则 |
| 错误处理 | 业务错误用 `thiserror::Error` 定义具体类型；禁止 `unwrap()`，禁止 `.expect("随便写点啥")`；允许 `.expect("明确的不可恢复错误")` |
| `unsafe` | 全局禁止；例外需在文档声明并代码 review |
| 日志 | 统一用 `tracing::info! / warn! / error! / debug!`；禁止 `println!` 进入生产代码 |
| 异步 | 全栈 `async/await` + tokio；禁止混合阻塞 IO 与异步（阻塞操作必须 `spawn_blocking`） |
| 集合 | 优先 `Vec`、`HashMap`；不可引入大集合做全局查找表（应放入 Resources） |
| 字符串 | 中文使用 UTF-8（协议内）；玩家名称等走 mir2-shared 编码工具统一处理 |
| 注释 | 对外 API（public fn）必须写 doc 注释（`///`）；复杂算法写行内说明；普通实现代码不写无意义废话 |

## 八、测试与 CI 规范

### 8.1 测试分层

| 层级 | 位置 | 示例 |
|------|------|------|
| 单元测试 | 同文件末尾 `#[cfg(test)] mod tests` | `calc::damage::tests` |
| 组件 / 逻辑验证 | `tests/` 目录 | 验证 DC=100 vs AC=30 的伤害输出在预期区间 |
| 集成测试 | `tests/` 目录 | 模拟玩家登录→移动→攻击→死亡全链路 |
| 一致性测试（重点） | `tests/consistency/` | **与经典 C# Random 算法同 seed 产生相同随机序列**；**A* 寻路与经典实现同起点同终点同路径** |

### 8.2 必写的一致性测试

- 伤害公式一致性：用 N 组 (DC, AC, 技能) → 输出结果必须在允许的小误差内与经典 C# 实现一致
- 经验升级曲线：从 1 级升到最高级每一级所需经验一致
- 随机数序列：相同 seed 产生的前 1000 个随机数必须字节对齐经典实现
- 寻路：选取 N 个典型地图点，Rust 版本路径长度与经典实现一致

### 8.3 CI 流水线（必须至少有这几步）

```
push / PR
  └─ cargo fmt --check           ← 格式错误 = 不通过
  └─ cargo clippy --all-targets --all-features -- -D warnings
  └─ cargo test --all            ← 全单元 + 集成 + 一致性测试
  └─ cargo build --release       ← 最终产物构建（可选，可放到 release 分支）
```

## 九、提交前自检清单（代码层面）

在你准备 `git commit` 之前，按顺序检查：

| # | 检查项 | 标准 |
|---|--------|------|
| 1 | 代码放在哪个 crate？是否符合 §3/§4/§6 的目录结构？ | 放错 crate = 拒绝提交 |
| 2 | 新写的数据结构是否落入 mir2-shared（而非本地私有）？ | 可共享的必须共享 |
| 3 | 游戏逻辑是否走 ECS（Component + System）？ | 禁止面向对象类继承 |
| 4 | Protobuf 字段是否新增编号（未重排/复用旧编号）？ | 协议版本兼容 |
| 5 | 是否信任客户端传值？有没有服务端独立校验？ | 移动/攻击/物品/金币 必须服务端重算 |
| 6 | 是否写了测试（至少覆盖关键算法）？ | 公式/随机/寻路必须有一致性测试 |
| 7 | `cargo fmt` + `cargo clippy --all-targets` 通过？ | 0 warning |
| 8 | 日志用 `tracing!`，无 `println!` 残留？ | 生产代码禁止裸打印 |
| 9 | 错误类型是具体 enum（`thiserror`），不滥用 `anyhow` 进业务 API？ | 对外返回有意义的错误码 |
| 10 | 是否触碰经典玩法数值（DC/MC/SC/AC/MAC/经验/掉落）？若是，有没有对照 V4.0 文档验证？ | 禁止凭空改写经典数值 |

**任意一项无法清晰回答 = 不要提交。先回到文档中找到对应章节，确认设计后再写。**

## 十、踩坑/反模式（看到这些写法 = 自动标记为可疑）

| 反模式 | 危害 | 正确替代 |
|--------|------|---------|
| `static mut GLOBAL_STATE: ...` | 并发不安全、不可测试、难调试 | 放入 ECS `Resources` 或 `tokio::sync::Mutex` |
| Player 结构体里写 `fn attack(target)` | 状态与行为耦合，后续扩展代价高 | 拆成 `CombatStats` / `TargetEntity` 等组件 + 独立 `combat_system` |
| 在 System 里直接 `socket.send()` | 业务与网络耦合，无回放/测试能力 | 写进事件队列，tick 结束后统一发送 |
| `unwrap()` / `.expect("todo")` | 生产环境偶发崩溃 | 正确的 `Result` 传播，或显式说明不可恢复错误 |
| 手写二进制协议不通过 protobuf | 协议变更难管理，前后端兼容难 | 统一 Protobuf + prost |
| 把"活动逻辑/对话逻辑"写死在 Rust 代码 | 每次改活动都要重新编译+发布+停服 | Lua 脚本，可热更新 |
| 在多个服务里写各自的 SQL（绕开 DBServer） | 不可审计/不可合服/不可调优 | 一律走 DBServer Repository |
| 客户端本地重写一套伤害公式 | 与服务端计算结果不一致，玩家看到的数值是错的 | 调用 mir2-shared 的 `calc::damage` |
| 每次写代码前**没有**读 V4.0 文档和本规范 | 架构漂移/重复劳动/返工 | 开工前通读一次 |
